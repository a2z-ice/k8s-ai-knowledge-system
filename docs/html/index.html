How we built qwen3:14b-k8s — a domain-specific Kubernetes assistant — and how you can customize any Ollama model for your own use case.

00 Why Customize? The Problem and the Proof

The Problem

A base LLM like qwen3:14b is general-purpose. When you ask it a Kubernetes question, it responds like a textbook — verbose explanations, code blocks, markdown headers, unpredictable formatting. This makes it unusable for programmatic consumption by tools like n8n agents.

Before vs After — Same Query, Same Weights, Different Config

01 What is Ollama Model Customization?

💡 Key Insight

Ollama model customization is NOT fine-tuning or retraining. The model weights (14 billion parameters) remain identical to the base model. You create a thin configuration wrapper — like a Docker image that layers config on top of a base image.

What ChangesWhat Stays the Same

Inference parameters (temperature, top_k)Tokenizer vocabulary (~152K tokens)

Disk space: The custom model manifest is a tiny JSON file (~1KB) pointing to the same 9.3GB weight blob as the base model. Ollama deduplicates at the blob level.

02 How LLM Inference Works

Tokenization

Text is split into subword tokens — the model only sees integer IDs:

Input:  "Kubernetes Deployment named coredns"
Tokens: ["Kuber", "netes", " Deploy", "ment", " named", " core", "dns"]
IDs:    [48291,   17352,   43891,    1326,   7086,    6254,   12847]

The model generates one token at a time. Each step, it outputs a probability distribution over its entire ~152,000 token vocabulary, then selects the next token based on the sampling strategy.

The KV Cache

The num_ctx parameter (32768 in our model) determines the KV cache size — how many tokens the model can "remember" during a single conversation. Larger context = more memory but can hold more tool results.

⚠️ Memory Impact

num_ctx=32768 with a 14B model requires approximately 14 GB RAM. Reduce to 8192 if memory is limited (~10 GB).

04 The Modelfile — Ollama's Dockerfile

A Modelfile is to Ollama what a Dockerfile is to Docker: a declarative recipe for building a model image.

# Base model (required)

# System prompt — injected before every conversation
SYSTEM """
Your instructions here
"""

# Inference parameters
PARAMETER temperature 0
PARAMETER num_ctx 32768
PARAMETER num_predict 1024
PARAMETER top_k 20
PARAMETER top_p 0.95
PARAMETER repeat_penalty 1
PARAMETER stop <|im_start|>
PARAMETER stop <|im_end|>

# Optional: LoRA fine-tune adapter weights
# ADAPTER ./my-lora.gguf

05 Step-by-Step: Building qwen3:14b-k8s

Pull the Base Model

ollama pull qwen3:14b
# Downloads ~9.3 GB on first run

Choose your base model from ollama.com/library. We use qwen3:14b for its excellent instruction following, tool-calling support, and 32K context window.

Create the Modelfile

cat > Modelfile.k8s << 'EOF'
FROM qwen3:14b

SYSTEM """
/no_think
You are a concise Kubernetes cluster assistant.
STRICT OUTPUT RULES:
- Output ONLY markdown tables. Nothing else.
- For counting queries: | Namespace | Count |
- For listing queries: | Kind | Name | Namespace | Details |
- NEVER output section headers (###), bullet lists, recommendations, or verbose text.
- NEVER hallucinate or invent resource names. Only use data from tool results.
- Maximum one summary sentence after the table.
"""

PARAMETER temperature 0
PARAMETER num_ctx 32768
PARAMETER num_predict 1024
PARAMETER top_k 20
PARAMETER top_p 0.95
PARAMETER repeat_penalty 1
PARAMETER stop <|im_start|>
PARAMETER stop <|im_end|>
EOF

Build the Custom Model

ollama create qwen3:14b-k8s -f Modelfile.k8s

06 Every Modelfile Directive Explained

DirectiveRequiredPurpose
FROMYesBase model (from Ollama library, local GGUF, or another custom model)
SYSTEMNoSystem prompt — persona, rules, output format (prepended to every conversation)
TEMPLATENoChat template (Go text/template). Usually inherited from base — rarely need to customize
PARAMETERNoInference parameters (temperature, num_ctx, stop tokens, etc.)
ADAPTERNoLoRA adapter weights (GGUF) — this is the only directive that modifies model behavior at the weight level
LICENSENoLicense metadata (doesn't affect inference)

The /no_think Directive

Qwen3 has a built-in chain-of-thought mode. Without /no_think, it wraps responses in <think>...</think> blocks:

Without /no_think

<think>
The user is asking about
deployments. I should query
the vector store...
The results show 3 deployments...
I should format as a table...
</think>

| Name | Namespace | Replicas |
|------|-----------|----------|
| coredns | kube-system | 2 |

With /no_think

| Name | Namespace | Replicas |
|------|-----------|----------|
| coredns | kube-system | 2 |
| kafka | k8s-ai | 1 |
| qdrant | k8s-ai | 1 |

We apply /no_think in three layers for maximum reliability:

Ollama Connection from Kubernetes Pods

Ollama runs on the host machine (never in a pod). Pods reach it via hostAliases:

# n8n-deployment.yaml
hostAliases:
  - ip: "192.168.1.154"        # Host machine's LAN IP
    hostnames:
      - "host.docker.internal"

# n8n credential (stored in sqlite DB)

  "id": "ollama-local",
  "type": "ollamaApi",
  "data": { "baseUrl": "http://host.docker.internal:11434" }

The think:false Runtime Patch

n8n's Ollama Chat Model node doesn't expose Qwen3's think parameter. We patch it at container startup:

# In n8n-deployment.yaml startup command:
OLLAMA_FILE=$(find /usr/local/lib/node_modules/n8n \
  -path "*/LMChatOllama/LmChatOllama.node.js" 2>/dev/null | head -1)

sed -i "s/const model = new ollama_1.ChatOllama({/\
  const model = new ollama_1.ChatOllama({ think: false,/" "$OLLAMA_FILE"

15 Line-by-Line Modelfile Explained

Every line of models/Modelfile.k8s serves a specific purpose. This section explains what each line does technically and what observable change it produces.

Line 1: FROM qwen3:14b

What it does: Specifies the base model. Ollama resolves this to a 9.3 GB GGUF weight blob on disk. The custom model's manifest points to this same blob — no duplication.

Why qwen3:14b:

  • 14B parameters — large enough for accurate tool-calling, small enough for 16 GB+ RAM
  • Native function/tool-calling support (critical for n8n AI Agent)
  • ChatML template format (<|im_start|> / <|im_end|>) — our stop tokens rely on this
  • Supports /no_think to suppress chain-of-thought

Observable outcome: Changing the model (e.g., FROM llama3.1:8b) changes the intelligence, capabilities, and template format entirely.

Line 4: /no_think

What it does: A Qwen3-specific directive that suppresses internal chain-of-thought reasoning. By default, Qwen3 wraps every response in <think>...</think> blocks containing 300-500 words of internal reasoning.

Without /no_think:
<think>
The user is asking about namespaces.
Let me think about what namespaces
exist in a default cluster...
[300-500 more words of reasoning]
</think>

Here are the default namespaces:
| Namespace | Purpose |
|-----------|---------|
| default | ... |
With /no_think:
| Namespace | Purpose |
|-----------|---------|
| default | Default namespace |
| kube-system | System components |
| kube-public | Public resources |
| kube-node-lease | Node heartbeats |
Highest-Impact Line

Response time drops from ~15s to ~4s. Output shrinks from 500+ tokens to ~50 tokens. This is the single most impactful line in the entire Modelfile.

Line 5: You are a concise Kubernetes cluster assistant.

What it does: Sets the model's persona. The word "concise" is critical — without it, the model defaults to verbose, helpful explanations. With it, responses are shorter and avoid unnecessary context.

Lines 6-7: STRICT OUTPUT RULES + Output ONLY markdown tables

What it does: "STRICT" and "RULES" signal mandatory compliance (LLMs respond more strongly to imperative framing). The table-only constraint eliminates prose, bullet lists, and headers.

Without rule:
Here are the namespaces:

### Namespaces
- **default** — User workloads
- **kube-system** — System pods
...
With rule:
| Namespace | Description |
|-----------|-------------|
| default | User workloads |
| kube-system | System pods |

Lines 8-9: Table Schema Templates

| Namespace | Count | for counting queries; | Kind | Name | Namespace | Details | for listing queries.

Observable outcome: Without explicit schemas, the model uses different column headers each time ("Resource" vs "Kind", sometimes including "Status" vs not). Fixed schemas make downstream parsing by n8n reliable and predictable.

Lines 10-12: Prohibitions and Grounding

LineRuleWhat It Prevents
10NEVER output headers, bullets, recommendationsModel adding ### headers above tables or "Recommendations:" sections below
11NEVER hallucinate or invent resource namesModel confidently listing pods like nginx-abc123 that don't exist in the cluster
12Maximum one summary sentence after the tableControlled relaxation — allows "4 namespaces found" without enabling verbose essays

Line 15: PARAMETER temperature 0

What it does: Greedy decoding — always pick the highest-probability token. No randomness.

temperature 0

Same input → identical output every time.
Critical for monitoring tools.

temperature 0.7 (default)

Same input → slightly different output each time.
Different column orders, wording variations.

Line 16: PARAMETER num_ctx 32768

What it does: 32K token context window. The kubernetes_inventory tool scrolls ALL Qdrant points (150+ resources × ~100 tokens each = 15,000 tokens). Plus system prompt, query, and history.

Observable outcome: With default num_ctx (2048-4096), tool results are silently truncated — "How many pods?" returns incomplete counts. At 32K, all resources fit. Trade-off: ~2 GB more VRAM.

Line 17: PARAMETER num_predict 1024

What it does: Caps output at 1,024 tokens (~750 words). Prevents runaway generation from malformed queries.

Observable outcome: Enough for a 50+ row table plus summary sentence. Without a cap, a bad query could generate thousands of tokens, consuming time and VRAM.

Lines 18-19: top_k 20 + top_p 0.95

What they do: Sampling safety nets. top_k=20 limits token candidates to top 20; top_p=0.95 eliminates the bottom 5% probability mass.

Observable outcome: At temperature 0, these have minimal practical effect (greedy always picks #1). They serve as safety nets — if temperature is ever changed to non-zero, these keep the vocabulary tight and prevent garbage tokens that break table formatting.

Line 20: PARAMETER repeat_penalty 1 (DISABLED)

What it does: Disables repetition penalty entirely. The default (1.1) penalizes tokens that already appeared, discouraging repetition.

Markdown tables repeat | on every row. With penalty enabled, the model avoids | after the first rows, silently corrupting all output. This is one of the most important parameters to get right.

Lines 21-22: PARAMETER stop <|im_start|> + stop <|im_end|>

What they do: ChatML control tokens. Generation halts immediately when either token is produced.

Without stop tokens: The model might hallucinate a fake follow-up — generating <|im_start|>user\nThank you!<|im_end|><|im_start|>assistant\nYou're welcome! as if the conversation continued.

With stop tokens: Clean termination at end of response. These are specific to Qwen3's ChatML format; other model families (LLaMA, Mistral) need different stop tokens.

16 Portability: Deploy Anywhere

💡 Key Principle

Your custom model is defined by a tiny Modelfile (~1 KB text file) + a base model from Ollama's registry. You never need to ship 9.3 GB of weights — just the Modelfile.

Method 1: Modelfile in Git (Recommended)

# On your dev machine — extract the Modelfile
ollama show qwen3:14b-k8s --modelfile > Modelfile.k8s
git add Modelfile.k8s && git commit -m "Add K8s model config"

# On ANY target machine (Mac, Linux, cloud VM):
ollama pull qwen3:14b                          # 9.3 GB download (one-time)
ollama create qwen3:14b-k8s -f Modelfile.k8s  # 40ms to build

Method 2: Docker Image (Kubernetes / Cloud)

FROM ollama/ollama:latest
COPY Modelfile.k8s /tmp/Modelfile.k8s
RUN ollama serve & sleep 5 && \
    ollama pull qwen3:14b && \
    ollama create qwen3:14b-k8s -f /tmp/Modelfile.k8s && \
    kill %1
EXPOSE 11434
CMD ["serve"]

Method 3: Offline / Air-Gapped

# Source machine: package everything
tar czf ollama-models.tar.gz ~/.ollama/models/

# Target machine: extract and run
tar xzf ollama-models.tar.gz -C ~/
ollama list  # qwen3:14b-k8s appears immediately
