About This System

A self-hosted AI assistant that continuously indexes a live Kubernetes cluster into a Qdrant vector database via Change Data Capture, and exposes a natural-language query interface through n8n and Ollama — entirely local, no cloud APIs required.

06 Workflow 1: CDC Pipeline (Real-Time Sync)

Purpose

Continuously synchronizes Kubernetes cluster state into the Qdrant vector database. When any of the 10 watched resource types is created, updated, or deleted, the change flows through Kafka into Qdrant within seconds.

Workflow ID: k8sCDCflow00001  |  Trigger: Kafka topic k8s-resources

Flow Diagram

Kafka Trigger (topic: k8s-resources, group: n8n-cdc-consumer)
    | raw JSON message from k8s-watcher
    v
Parse Message Code
    | extract event_type, resource_uid, kind, name, namespace, embed_text
    v
Is Delete Event? If/Else
    |                       |
    | TRUE                  | FALSE
    v                       v
Delete Vector HTTP  Format Document Code
    (filter by                  | build pageContent + metadata
     resource_uid)              v
    [STOP]           Generate Embedding HTTP

                                | returns 768-dim vector
                                v

                                PUT /collections/k8s/points

Node-by-Node Encyclopedia

1. Kafka Trigger n8n-nodes-base.kafkaTrigger v1
Listens for new messages on the k8s-resources Kafka topic
SettingValueWhy
Topick8s-resourcesSingle topic for all 10 resource types
Group IDn8n-cdc-consumerConsumer group for offset tracking
Auto Offset ResetlatestOnly consume NEW messages; prevents replay of history on restart
Auto Topic CreationfalseTopic must be pre-created by Kafka
Critical: autoOffsetReset must be "latest"

Changing to "earliest" causes Kafka to replay the entire topic history on every n8n restart, flooding Qdrant with stale/duplicate points and breaking the Reset E2E test.

2. Parse Message n8n-nodes-base.code v2 (JavaScript)
Extracts JSON payload from Kafka message and ensures embed_text field exists

Value: Converts raw Kafka bytes into structured data. If k8s-watcher already included embed_text, uses it directly. Otherwise, constructs natural-language embed text: "Kubernetes {kind} named {name} in namespace {ns}. Labels: ..."

Customization tip: To change how resources are described to the AI, modify the embed_text construction in this node. Natural language format gives significantly better cosine similarity than terse key-value formats.

3. Is Delete Event? n8n-nodes-base.if v2
Routes DELETED events to cleanup path; all other events continue to upsert path

Condition: event_type === "DELETED" (string equals, case-sensitive)

Value: Prevents inserting a vector for a resource that no longer exists. DELETED events only need to remove the old Qdrant point.

4. Delete Existing Vector n8n-nodes-base.httpRequest v4
Removes the previous Qdrant point for this resource before inserting the updated version

Endpoint: POST http://qdrant:6333/collections/k8s/points/delete

Body: Filter by metadata.resource_uid — finds both test-inserted and CDC-inserted points.

Value: Ensures deduplication. Without this, every MODIFIED event would create a duplicate vector. The filter-delete approach works regardless of how the point was originally inserted (by point ID or by auto-generated UUID).

5. Format Document n8n-nodes-base.code v2 (JavaScript)
Structures the payload with pageContent (for embedding) and metadata (for Qdrant storage)

Value: Creates a compact spec_summary instead of storing the entire raw_spec_json. Includes resource-type-specific parsing: container images for Deployments, port mappings for Services, storage for PVCs, key names (never values) for Secrets.

Customization tip: To add custom metadata fields to Qdrant points, modify the metadata object in this node. Every field here becomes searchable/filterable in Qdrant.

6. Generate Embedding n8n-nodes-base.httpRequest v4
Calls Ollama to generate a 768-dimensional vector from the embed_text

Endpoint: POST http://host.docker.internal:11434/api/embeddings

Model: nomic-embed-text:latest

Retry: 3 attempts with 2s delay (handles Ollama queue contention)

Batching: 1 message per batch, 1s interval (prevents Ollama overload)

Value: Converts natural-language resource descriptions into numerical vectors that enable semantic search. The 768-dim nomic-embed-text model produces scores in the 0.38–0.70 range for Kubernetes metadata.

7. Upsert to Qdrant n8n-nodes-base.httpRequest v4
Inserts the vector + metadata into the Qdrant k8s collection

Endpoint: PUT http://qdrant:6333/collections/k8s/points

Point ID: resource_uid (Kubernetes metadata.uid — deterministic, enables idempotent upserts)

Retry: 3 attempts with 2s delay

Value: Uses HTTP PUT (not the n8n Qdrant node) because the native node auto-generates UUID point IDs and cannot use resource_uid. HTTP PUT enables idempotent upserts — same UID always overwrites the same point.

System Prompt (Critical for Behavior)

The AI Agent's system message defines HOW the agent selects tools and formats responses. This is the second layer of the two-layer prompt system (Modelfile SYSTEM is layer 1):

You are a Kubernetes cluster monitoring assistant for the kind cluster "k8s-ai".

/no_think

You have TWO tools. You MUST call a tool before EVERY answer.

## Tool Selection

**kubernetes_inventory** — ALWAYS use this when:
- Counting ("how many pods?")
- Listing ALL resources of a type ("list all configmaps")
- Aggregation ("resources per namespace")
- Follow-up about previously counted resources
- Cluster overview ("what is running?")

**kubernetes_search** — Use ONLY when:
- ONE specific named resource ("show me the coredns deployment")
- Searching by label ("find services with app=kafka")
- Deep details about a single resource

## Critical Rules
1. ALWAYS call a tool before answering
2. When in doubt, use kubernetes_inventory
3. NEVER invent or hallucinate resources
4. Output concise markdown tables
5. Never expose resource_uid or secret values

Customization tip: To add new tools or change tool selection logic, modify this system message. The model reads these rules on every query.

4. Qdrant Vector Store (kubernetes_search) @n8n/n8n-nodes-langchain.vectorStoreQdrant v1
Semantic search tool — embeds the query and finds the most similar K8s resources in Qdrant
SettingValueWhy

Value: Enables the AI to answer questions about specific resources ("tell me about coredns") by finding semantically similar vectors. The score threshold of 0.3 ensures Deployments (which score ~0.43) aren't silently excluded.

Customization tip: Increase Top K for larger clusters. Lower the score threshold if resources are being missed. To add a second collection (e.g., for logs), add another Vector Store node.

Value: Converts the user's natural-language query into a vector that can be compared against stored resource vectors using cosine similarity.

6. Kubernetes Inventory Tool (Code Tool) @n8n/n8n-nodes-langchain.toolCode v1
Scrolls ALL Qdrant points and returns a complete inventory grouped by kind and namespace
SettingValueWhy
Tool Namekubernetes_inventoryHow the Agent refers to this tool
LanguageJavaScriptRuns inside n8n's sandbox

How it works:

  1. Paginates through ALL Qdrant points (100 per page) using this.helpers.httpRequest()
  2. Extracts kind, name, namespace, labels, spec from each point's metadata
  3. Groups resources by namespace, then by kind within each namespace
  4. Returns a formatted text with namespace summaries and per-kind breakdowns

Value: Enables accurate counting and listing queries. The kubernetes_search tool (vector search) only returns top-20 matches — it can't count ALL pods. This tool scrolls everything and gives complete answers.

Why Code Tool instead of Workflow Tool: The toolWorkflow node requires complex workflowInputs resourceMapper config that fails on JSON import. toolCode with this.helpers.httpRequest() is simpler and reliable.

Customization tip: To add custom grouping (e.g., by label), modify the JavaScript code. To add new data sources, add more HTTP calls.

7. Postgres Chat Memory @n8n/n8n-nodes-langchain.memoryPostgresChat v1.3

Customization tip: For multi-user support, change sessionKey to a dynamic value (e.g., user ID from the request). Increase contextWindowLength for longer conversations (costs more tokens).

08 Workflow 3: Reset & Resync

Purpose

Wipes the entire Qdrant vector database and triggers a full re-index from the live Kubernetes cluster. Use this when vectors are stale, corrupted, or after schema changes.

Workflow ID: k8sRSTflow00001  |  Triggers: POST /webhook/k8s-reset + Manual Trigger

Reset Webhook ——————+

                           v
                  Delete Qdrant Collection HTTP DELETE
                  (http://qdrant:6333/collections/k8s)
                           |
                           v
                  Recreate Qdrant Collection HTTP PUT

                           |

Node-by-Node Encyclopedia

1. Reset WebhookWebhook v1
Accepts POST requests to trigger the reset pipeline

Endpoint: POST http://localhost:30000/webhook/k8s-reset

Response Mode: lastNode — returns the Format Response output as the HTTP response

2. Delete Qdrant CollectionHTTP Request v4
Drops the entire k8s collection — all vectors and metadata removed

URL: DELETE http://qdrant:6333/collections/k8s

Value: Clean slate. Faster than deleting points one-by-one.

3. Recreate Qdrant CollectionHTTP Request v4
Creates a fresh k8s collection with the correct vector schema

URL: PUT http://qdrant:6333/collections/k8s

Schema: 768 dimensions, Cosine distance, 2 segments, replication factor 1

Value: Ensures the collection schema matches what the embedding model produces (768-dim for nomic-embed-text).

4. Trigger ResyncHTTP Request v4
Tells k8s-watcher to re-enumerate all cluster resources and publish them to Kafka

URL: POST http://k8s-watcher:8080/resync

Behavior: Returns 202 Accepted immediately. Resync runs asynchronously in a background thread. Qdrant repopulates in ~30-45 seconds.

5. Format ResponseCode v2
Returns a JSON status object as the webhook response
{ "status": "ok", "message": "Qdrant collection cleared and resync triggered...", "reset_at": "2026-03-12T..." }
After Reset: Wait 30-45 Seconds

Qdrant is empty immediately after reset while k8s-watcher republishes all resources. Do not run queries or E2E tests until points_count ≥ 10.

Security: Secret Values Never Exposed

The system indexes Secret metadata only (name, type, key names). Base64-encoded values are discarded by k8s-watcher before publishing to Kafka. The AI can tell you "Secret X has keys: username, password" but can never reveal the actual values.

17 k8s-watcher: Kubernetes API Monitor

A Python service running as a pod in the k8s-ai namespace. Watches the Kubernetes API for changes to 10 resource types and publishes events to Kafka.

# {"status":"ok"}

  "last_updated_timestamp": "2026-03-12T..."
}

Secret Safety

Secret Values Never Leave the Cluster

When kind == "Secret", the watcher replaces the spec with {"type": "Opaque", "dataKeys": ["username", "password"]}. The base64-encoded values in raw["data"] are discarded. This applies to both watch events and resync.

Customization: Adding New Resource Types

To watch additional resources (e.g., Ingress, CronJob), edit k8s-watcher/watcher.py:

  1. Add a new watch thread in the main() function
  2. Add the list function to resync_all()
  3. Rebuild: docker build -t k8s-watcher:latest ./k8s-watcher/ && kind load docker-image k8s-watcher:latest --name k8s-ai
  4. Restart: kubectl -n k8s-ai rollout restart deployment/k8s-watcher
