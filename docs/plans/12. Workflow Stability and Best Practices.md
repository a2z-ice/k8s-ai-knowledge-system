## Workflow 1: CDC_K8s_Flow (`k8sCDCflow00001`)

### 1A. Error Output Routing on Generate Embedding + Upsert to Qdrant

| Aspect | Before | After |
|--------|--------|-------|
| **Generate Embedding** `onError` | _(not set — default: stop execution)_ | `"continueErrorOutput"` |
| **Upsert to Qdrant** `onError` | _(not set — default: stop execution)_ | `"continueErrorOutput"` |
| **Error sink** | None — retries exhausted → silent drop | New **Log CDC Error** Code node at `[1520, 600]` |
| **Error connections** | None | Generate Embedding output index 1 → Log CDC Error; Upsert to Qdrant output index 1 → Log CDC Error |
| **Log output** | Nothing visible in `kubectl logs` | `CDC_ERROR: Failed to process {kind}/{name} (uid={uid}): {error}` on stderr |

**Why:** When Ollama is temporarily unavailable or Qdrant rejects a point, the resource was silently lost after 3 retries. Now the error is logged to stdout (visible in `kubectl -n k8s-ai logs deployment/n8n`) and the pipeline continues processing the next Kafka message.

### 1B. Required-Fields Validation in Parse Message

| Aspect | Before | After |
|--------|--------|-------|
| **Validation** | None — malformed Kafka messages pass through | Checks `resource_uid`, `kind`, `name` are truthy |
| **On failure** | Downstream nodes receive incomplete data → unpredictable errors | Returns `[]` (empty array) → skips all downstream nodes |
| **Logging** | None | `CDC_SKIP: Missing required fields: uid=..., kind=..., name=...` on stderr |

**Code added** (after embed_text construction in Parse Message `jsCode`):
```javascript
if (!data.resource_uid || !data.kind || !data.name) {
  console.error(`CDC_SKIP: Missing required fields: uid=${data.resource_uid}, kind=${data.kind}, name=${data.name}`);
  return [];  // empty = skip downstream nodes
}
```

### 1C. Batching on Upsert to Qdrant

| Aspect | Before | After |
|--------|--------|-------|
| **Upsert batching** | No batching configured | `batchSize: 1, batchInterval: 500ms` |

**Why:** During resync bursts (k8s-watcher republishes all ~75 resources), unbatched upserts can overwhelm Qdrant. The 500ms interval spaces writes out, preventing timeouts.

**Note:** Generate Embedding already had `batchSize: 1, batchInterval: 1000ms` — unchanged.

### 1D. Delete Existing Vector Hardening

| Aspect | Before | After |
|--------|--------|-------|
| **`retryOnFail`** | `false` (default) | `true` |
| **`maxTries`** | 1 (default) | `2` |
| **`waitBetweenTries`** | 0 | `1000ms` |
| **`timeout`** | None (infinite wait) | `10000ms` (10s) |
| **`onError`** | _(default: stop execution)_ | `"continueRegularOutput"` — if point doesn't exist, continue gracefully |

**Why:** The Delete node fires for every CDC event (both updates and deletes). If Qdrant is temporarily slow, no retry meant the entire pipeline stopped. With `continueRegularOutput`, a 404 (point already gone) or timeout doesn't block the upsert path.

## Workflow 2: AI_K8s_Flow (`k8sAIflow000001`)

### 2A. topK Increase

| Aspect | Before | After |
|--------|--------|-------|
| **Qdrant Vector Store `topK`** | `20` | `30` |

**Why:** With compact `spec_summary` payloads (~200 bytes/result vs ~5KB with full `raw_spec_json`), 30 results fit easily in qwen3:14b-k8s's 32K context window. Improves recall for clusters with 50+ resources.

### 2B. try/catch in Kubernetes Inventory Tool

| Aspect | Before | After |
|--------|--------|-------|
| **Scroll loop error handling** | None — unhandled exception crashes the tool | `try/catch` wrapping the entire pagination loop |
| **Agent receives** | Raw error object (unparseable) | Descriptive string: `"Error: Unable to fetch Kubernetes inventory. The vector database may be temporarily unavailable. Details: {message}"` |

**Code change** (in Kubernetes Inventory Tool `jsCode`):
```javascript
// BEFORE:
let allPoints = [];
let offset = null;
do {
  // ... pagination loop ...
} while (offset);

// AFTER:
let allPoints = [];
let offset = null;
try {
  do {
    // ... pagination loop ...
  } while (offset);
} catch (err) {
  return `Error: Unable to fetch Kubernetes inventory. The vector database may be temporarily unavailable. Details: ${err.message || err}`;
}
```

**Why:** If Qdrant is down (e.g., during a reset), the tool threw an unhandled error that the AI Agent couldn't interpret. Now the agent receives a human-readable message and can inform the user.

### 2C. Empty-Input Guard in System Prompt

| Aspect | Before | After |
|--------|--------|-------|
| **System prompt rules** | 9 rules | 10 rules |
| **Rule 10** | _(not present)_ | `"If the user sends an empty or meaningless message, respond: 'Please ask a specific question about your Kubernetes cluster.'"` |

**Why:** Empty or garbage inputs waste a tool call + Ollama inference. A system prompt guard is safer than an IF node (which would break the langchain chatTrigger → Agent data flow).

## Workflow 3: Reset_K8s_Flow (`k8sRSTflow00001`)

### 3A. Retry + Timeout on All HTTP Nodes

| Node | Before | After |
|------|--------|-------|
| **Delete Qdrant Collection** | No retry, no timeout | `retryOnFail: true, maxTries: 2, waitBetweenTries: 2000, timeout: 10000` |
| **Recreate Qdrant Collection** | No retry, no timeout | `retryOnFail: true, maxTries: 3, waitBetweenTries: 2000, timeout: 10000` |
| **Trigger Resync** | No retry, no timeout | `retryOnFail: true, maxTries: 2, waitBetweenTries: 3000, timeout: 15000` |

**Why:** The Reset flow is a critical recovery operation. All 3 HTTP nodes had `"options": {}` with no retry or timeout. A transient network blip between n8n and Qdrant (both in-cluster) would fail the entire reset. Recreate gets 3 tries because it's the most critical step (a failed recreate leaves no collection).

### 3B. Graceful Delete Failure

| Aspect | Before | After |
|--------|--------|-------|
| **Delete Qdrant Collection `onError`** | _(default: stop execution)_ | `"continueRegularOutput"` |

**Why:** On first run or if collection was already deleted, the DELETE returns 404. Without `continueRegularOutput`, the entire reset chain stops. Now it continues to Recreate.

### 3C. Enhanced Format Response

| Aspect | Before | After |
|--------|--------|-------|
| **Response fields** | `{status, message, reset_at}` | `{status, message, reset_at, resync_response}` |
| **Resync acknowledgment** | None — no way to confirm resync was triggered | `resync_response` field shows the k8s-watcher response status |

**Code change:**
```javascript
// BEFORE:
const ts = new Date().toISOString();
return [{ json: { status: 'ok', message: '...', reset_at: ts } }];

// AFTER:
const ts = new Date().toISOString();
const resyncResult = $('Trigger Resync').first().json;
return [{ json: {
  status: 'ok',
  message: 'Qdrant collection cleared and k8s-watcher resync triggered. Vector database will repopulate within ~30 seconds.',
  reset_at: ts,
  resync_response: resyncResult?.status || 'triggered'
} }];
```

**Test compatibility:** E2E test 15 checks `body.status === 'ok'` and `body.reset_at` — both preserved. The new `resync_response` field is additive.

## Workflow 4: Memory_Clear_Flow (`k8sMEMclear001`)

### 4A. Count → Delete → Format Response Chain

| Aspect | Before | After |
|--------|--------|-------|
| **Node chain** | Triggers → Clear Memory (bare DELETE) | Triggers → **Count Memory Rows** → Clear Memory → **Format Response** |
| **Count node** | _(not present)_ | Postgres: `SELECT COUNT(*) as row_count FROM n8n_chat_histories;` |
| **Format Response** | _(not present)_ | Code node: returns `{status, rows_deleted, cleared_at}`, logs `MEMORY_CLEAR: Deleted N rows at timestamp` |
| **Operational visibility** | No output, no logging | Row count logged to stdout + structured JSON response |

**Node layout:**
```
Before:  Manual Trigger ──┐
                          ├──→ Clear Memory
         Schedule Trigger ─┘

After:   Manual Trigger ──┐
                          ├──→ Count Memory Rows → Clear Memory → Format Response
         Schedule Trigger ─┘
```

### 4B. Error Handling on Count Node

| Aspect | Before | After |
|--------|--------|-------|
| **Count node `onError`** | _(not applicable — node didn't exist)_ | `"continueRegularOutput"` |

**Why:** If the `n8n_chat_histories` table doesn't exist yet (fresh Postgres), the COUNT query fails. With `continueRegularOutput`, the flow continues to DELETE (which also gracefully handles missing tables) and Format Response reports 0 rows.

## Bug Fix: `scripts/setup.sh` — `n8n_exec()` User Context

### Problem

The n8n container runs as root (`securityContext.runAsUser: 0`) but the n8n server process drops to user `node` (uid 1000) via `exec su -s /bin/sh node -c "exec /docker-entrypoint.sh"`. The server's data directory is `/home/node/.n8n` (PVC-mounted).

When `kubectl exec` runs `n8n import:workflow`, it executes as root. The n8n CLI defaults to `~/.n8n` = `/root/.n8n` — a **separate, empty database**. The import succeeds (creating a fresh SQLite DB with all migrations at `/root/.n8n/database.sqlite`), but the running n8n server never sees these workflows because it reads from `/home/node/.n8n/database.sqlite`.

**Symptom:** `setup.sh --keep-cluster` completes successfully, imports report "Successfully imported", verify step lists correct IDs, but webhooks return HTTP 404. n8n logs show `"Could not find workflow with id ..."`.

### Fix

| Aspect | Before | After |
|--------|--------|-------|
| **`n8n_exec()` function** | `kubectl exec "${pod}" -- "$@"` | `kubectl exec "${pod}" -- su -s /bin/sh node -c "$*"` |
| **CLI runs as** | root (uid 0) | node (uid 1000) |
| **CLI data directory** | `/root/.n8n` (wrong DB) | `/home/node/.n8n` (correct DB) |

**Code change in `scripts/setup.sh`:**
```bash
# BEFORE:
n8n_exec() {
  local pod
  pod="$(get_n8n_pod)"
  kubectl --context "${CONTEXT}" -n "${NAMESPACE}" exec "${pod}" -- "$@"
}

# AFTER:
n8n_exec() {
  local pod
  pod="$(get_n8n_pod)"
  # Run as user 'node' so the CLI uses /home/node/.n8n (same DB as the server).
  # The container runs as root but the n8n server drops to uid 1000 (node).
  kubectl --context "${CONTEXT}" -n "${NAMESPACE}" exec "${pod}" -- su -s /bin/sh node -c "$*"
}
```

**Note:** The `n8n export:workflow --all` command also needs this fix to export from the correct DB. The verify step in setup.sh (which exports all workflows to check IDs) now correctly reads the server's DB.

## Test Fix: E2E Test 9 — Secrets Query Assertion

### Problem

Test 9 asserts `expect(answer.toLowerCase()).toMatch(/secret/)`. The AI now returns concise markdown tables (following system prompt rules 7-8) like:

```
| namespace   | name                         | type                              |
|-------------|------------------------------|-----------------------------------|
| kube-system | bootstrap-token-abcdef       | bootstrap.kubernetes.io/token     |
```

The word "secret" doesn't appear anywhere — column headers are "namespace", "name", "type" and values contain "bootstrap.kubernetes.io/token" but not "secret".

### Fix

```javascript
// BEFORE:
expect(answer.toLowerCase()).toMatch(/secret/);

// AFTER:
// The AI may return a concise table without the literal word "secret" —
// accept the bootstrap token type as proof that secret metadata was returned
expect(answer.toLowerCase()).toMatch(/secret|bootstrap\.kubernetes\.io/);
```

The `bootstrap.kubernetes.io/token` type value is unique to Kubernetes Secret resources and proves the AI correctly retrieved and returned secret metadata.
