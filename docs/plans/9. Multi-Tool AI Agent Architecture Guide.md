**Date:** 2026-03-08

## Purpose of This Document

This document explains step by step:
1. Why the original "Scroll All Resources → Build Inventory" approach was a shortcut
2. How the multi-tool AI Agent architecture works
3. What `kubernetes_inventory` does and why it exists
4. Why the Qdrant Vector Store (`kubernetes_search`) is called less frequently
5. How the AI Agent decides which tool to use

## The Problem: Semantic Search Cannot Count

The core problem that drove every architectural decision:

**Semantic search returns the TOP-K most similar results, not ALL results.**

Example: User asks "How many pods are in each namespace?"

If we use Qdrant Vector Store semantic search with topK=20:
- Query vector is embedded from "pods in each namespace"
- Qdrant returns the 20 most semantically similar points
- But these 20 might include Deployments, Services, ConfigMaps that mention "pod"
- Out of 15 actual pods, we might get only 8 pods + 12 other resource types
- The AI gives wrong counts

This is not a bug in Qdrant or n8n — it's how vector similarity search works. It finds *relevant* results, not *all* results of a type.

## Architecture 1: The Shortcut (Scroll + Build Inventory)

### What It Did

```
Chat Trigger
  │ (user asks "How many pods?")
  ▼
Scroll All Resources  ← HTTP POST to Qdrant scroll API (limit: 200)
  │ (returns ALL 80 resources as JSON)
  ▼
Build Inventory  ← Code node: groups by kind/namespace, builds text
  │ (outputs enriched chatInput with all data prepended)
  ▼
AI Agent  ← promptType: "define", text: "={{ $json.chatInput }}"
  ├── Qdrant Vector Store (kubernetes_search) ← NEVER CALLED
  ├── Ollama Chat Model
  └── Postgres Chat Memory
```

### Step-by-Step Execution

1. User sends: `"How many pods in each namespace?"`
2. Chat Trigger passes the message downstream
3. **Scroll All Resources** makes an HTTP POST to Qdrant:
   ```
   POST http://qdrant:6333/collections/k8s/points/scroll
   Body: { "limit": 200, "with_payload": { "include": ["metadata"] } }
   ```
   Returns all 80 points with metadata (kind, name, namespace, spec_summary).

4. **Build Inventory** runs JavaScript code that:
   - Groups all 80 resources by kind (Pod, Deployment, Service, etc.)
   - Within each kind, groups by namespace
   - Prepends this as `[CLUSTER INVENTORY]` to the original question
   - Outputs: `{ chatInput: "[CLUSTER INVENTORY - 80 resources total]\nPod (15 total):\n  k8s-ai (6): ...\n\n[USER QUESTION]\nHow many pods in each namespace?" }`

5. **AI Agent** receives this enriched chatInput via `promptType: "define"`. The LLM sees ALL resources in its prompt context.

6. The LLM reads the inventory text, counts pods per namespace, outputs a table. **It never calls any tool** because all data is already in the prompt.

### Why This Was Wrong

| Problem | Detail |
|---------|--------|
| **Qdrant Vector Store tool was useless** | Connected to the agent but never called. The agent had no reason to search — all data was pre-loaded. |
| **Every query loaded all 80 resources** | Even "what is the coredns deployment?" loaded everything. ~2KB of wasted context per query. |
| **`promptType: "define"` was a hack** | The n8n AI Agent with chatTrigger reads `chatInput` from the trigger's session, NOT from upstream nodes. Without this hack, the agent saw the original question only, not the enriched one. This override was a sign the architecture was wrong. |
| **Not scalable** | A cluster with 1000+ resources would overflow the context window. |
| **Not a real agent** | The AI Agent was just parsing pre-loaded text, not using tools intelligently. |
| **No tool selection intelligence** | The agent couldn't differentiate between "count all pods" (needs all data) and "show coredns spec" (needs one resource). Both queries loaded everything. |

## Architecture 2: Multi-Tool AI Agent (Current)

### What It Does

```
Chat Trigger
  │ (user asks "How many pods?")
  ▼
AI Agent  ← toolsAgent, maxIterations=5
  │
  │  The agent reads the system prompt and decides which tool to call:
  │
  ├── Tool 1: kubernetes_inventory (Code Tool)
  │     Purpose: Get ALL resources, grouped by kind/namespace with counts
  │     When used: Counting, listing all, aggregation queries
  │     How: Calls Qdrant scroll API via this.helpers.httpRequest()
  │
  ├── Tool 2: kubernetes_search (Qdrant Vector Store, native n8n node)
  │     Purpose: Find specific resources by semantic similarity
  │     When used: Detail queries about specific resources
  │     How: Embeds query → Qdrant cosine search → top 20 results
  │     └── Embeddings Ollama (nomic-embed-text, 768-dim)
  │
  ├── Ollama Chat Model (qwen3:14b-k8s, temperature=0)
  └── Postgres Chat Memory (session: k8s-ai-global, contextWindow=5)
```

### Step-by-Step: Aggregation Query

User asks: **"How many pods in each namespace?"**

**Step 1 — Chat Trigger receives the message**
- The user's raw question is passed directly to the AI Agent
- No pre-processing, no pre-loading, no Scroll node

**Step 2 — AI Agent reads the system prompt**
The system prompt says:
```
kubernetes_inventory — Call this for:
- Counting resources ("how many pods?")
- Listing ALL resources of a type
- Aggregation queries ("resources per namespace")

kubernetes_search — Call this for:
- Specific resource details ("show me the coredns deployment spec")
- Searching by name or label
```

**Step 3 — Agent decides: this is a counting query → call kubernetes_inventory**
The agent generates a tool call: `kubernetes_inventory("How many pods in each namespace?")`

**Step 4 — kubernetes_inventory Code Tool executes**
The JavaScript code runs inside the n8n sandbox:
```javascript
// 1. Call Qdrant scroll API to get ALL resources
const response = await this.helpers.httpRequest({
  method: 'POST',
  url: 'http://qdrant:6333/collections/k8s/points/scroll',
  body: { limit: 200, with_payload: { include: ['metadata'] } },
  json: true
});

// 2. Group by kind and namespace
const points = response.result?.points || [];
const byKind = {};
for (const p of points) {
  const m = p.payload?.metadata || {};
  const kind = m.kind || '?';
  // ... group by kind, then by namespace
}

// 3. Return structured inventory text
return lines.join('\n');
```

The tool returns:
```
CLUSTER INVENTORY — 80 resources total

ConfigMap (15 total):
  default (1): kube-root-ca.crt
  k8s-ai (2): pgadmin-servers, kube-root-ca.crt
  ...
Pod (15 total):
  k8s-ai (6): k8s-watcher-xxx, kafka-0, n8n-xxx, ...
  kube-system (8): coredns-xxx, etcd-xxx, ...
  local-path-storage (1): local-path-provisioner-xxx
...
```

**Step 5 — Agent reads the tool result and generates the answer**
The LLM sees the complete inventory with exact counts. It produces:
```
| Namespace          | Count |
|--------------------|-------|
| k8s-ai             | 6     |
| kube-system        | 8     |
| local-path-storage | 1     |
```

**Step 6 — Response returned to user**

### Step-by-Step: Detail Query

User asks: **"What labels does the coredns deployment have?"**

**Step 1–2** — Same as above. Chat Trigger → AI Agent reads system prompt.

**Step 3 — Agent decides: this is a specific resource query → call kubernetes_search**
The agent generates a tool call: `kubernetes_search("coredns deployment labels kube-system")`

**Step 4 — kubernetes_search (Qdrant Vector Store) executes**
This is the native n8n Qdrant Vector Store node in `retrieve-as-tool` mode:
1. The search query is embedded using **Embeddings Ollama** (nomic-embed-text) → 768-dim vector
2. Qdrant cosine similarity search runs with topK=20
3. Returns the 20 most similar resources with full metadata (labels, annotations, spec_summary)
4. The coredns Deployment ranks highest because its embed text contains "Kubernetes Deployment named coredns in namespace kube-system. Labels: k8s-app=kube-dns"

**Step 5 — Agent reads the search results**
The results include the coredns deployment with its labels field: `k8s-app=kube-dns`
The agent outputs a table showing the labels.

**Step 6 — Response returned to user**

## What Is kubernetes_inventory and Why Does It Exist?

### The Fundamental Need

Semantic search finds *similar* results. But many cluster monitoring questions need *all* results:
- "How many pods?" → need all pods, not just the most similar ones
- "List all namespaces" → need every namespace, not a relevance-ranked subset
- "Count resources per namespace" → need a complete count, not an approximation

### What kubernetes_inventory Does

It is a **Code Tool** (n8n `toolCode` node) that:

1. **Calls Qdrant's scroll API** — `POST /collections/k8s/points/scroll` with limit=200. This is NOT a semantic search — it returns ALL points in the collection, unranked.

2. **Groups results by kind and namespace** — Organizes the raw data into a human-readable summary:
   ```
   CLUSTER INVENTORY — 80 resources total

   ConfigMap (15 total):
     default (1): kube-root-ca.crt
     k8s-ai (2): pgadmin-servers, kube-root-ca.crt
   Deployment (7 total):
     k8s-ai (5): n8n, qdrant, pgadmin, k8s-watcher, postgres
     kube-system (1): coredns
     local-path-storage (1): local-path-provisioner
   Pod (15 total):
     k8s-ai (6): k8s-watcher-xxx, kafka-0, ...
     kube-system (8): coredns-xxx, etcd-xxx, ...
     local-path-storage (1): local-path-provisioner-xxx
   ```

3. **Returns this text to the AI Agent** — The agent reads it and answers the user's question.

### Why Code Tool (Not a Sub-Workflow or Native Node)

| Approach | Problem |
|----------|---------|
| **toolWorkflow** (sub-workflow tool) | Requires `workflowInputs` resourceMapper that's hard to configure via JSON import. Failed with "The workflow did not return a response" errors. |
| **Native Qdrant Vector Store** | Only does semantic search (requires a query vector). There is no "scroll all" mode. |
| **toolHttpRequest** | Raw HTTP response would be too large and unformatted for the LLM. |
| **toolCode with this.helpers.httpRequest()** | Works. Simple. Reliable. Code runs inline. No sub-workflow to activate. |

### Why HTTP Instead of Native Qdrant Node for Scrolling

The native n8n Qdrant Vector Store node has these modes:
- `retrieve-as-tool` — semantic search (needs a query vector, returns topK)
- `insert` — insert documents (auto-generates point IDs)
- `load` — load documents for downstream processing

**None of these modes support "scroll all documents"**. The scroll API is a Qdrant-specific operation that returns ALL points without any similarity ranking. The only way to call it from n8n is via HTTP Request.

## Why Qdrant Vector Store (kubernetes_search) Is Called Less Often

### The Usage Pattern

| Query Type | Tool Called | Why |
|-----------|-------------|-----|
| "How many pods?" | `kubernetes_inventory` | Needs ALL pods. Semantic search returns partial. |
| "List all deployments" | `kubernetes_inventory` | Needs ALL deployments. |
| "Resources per namespace" | `kubernetes_inventory` | Needs complete aggregation. |
| "What is the coredns spec?" | `kubernetes_search` | Needs ONE specific resource. |
| "Show services with app=kafka" | `kubernetes_search` | Targeted search by label. |
| "What labels does n8n have?" | `kubernetes_search` | Specific resource detail. |

**Most Kubernetes monitoring questions are aggregation queries** — "how many", "list all", "show everything". These all use `kubernetes_inventory`.

**Detail queries are less common** — "what is the spec of X", "what labels does Y have". These use `kubernetes_search`.

This is why you observe `kubernetes_search` (Qdrant Vector Store) being called less often. It's not broken — it's used for the right queries. The agent's system prompt correctly routes:
- Counting/listing → `kubernetes_inventory`
- Specific details → `kubernetes_search`

### When Would kubernetes_search Be Called More?

If users frequently asked questions like:
- "Tell me about the kafka statefulset"
- "What containers does the n8n deployment run?"
- "What ports does the qdrant service expose?"
- "Find resources with label app=postgres"

These are all specific detail queries that benefit from semantic search.

## Key Differences Between the Two Architectures

| Aspect | Shortcut (Scroll+Build) | Multi-Tool Agent |
|--------|------------------------|------------------|
| **Data loading** | ALL data loaded EVERY query | Data loaded only when agent calls a tool |
| **Tool usage** | Tools never called | Agent actively decides which tool |
| **Qdrant Vector Store** | Connected but useless | Called for detail queries |
| **kubernetes_inventory** | Didn't exist — Build Inventory was inline | Called for aggregation queries |
| **promptType hack** | Required (`promptType: "define"`) | Not needed — agent reads chatInput normally |
| **Scalability** | Breaks at ~200 resources | Scales — only loads data when needed |
| **Intelligence** | Text parser | Tool-using agent with decision logic |
| **maxIterations** | 2 (never needed more) | 5 (may call both tools) |

## How the System Prompt Controls Tool Selection

The system prompt is the most critical piece. Without it, the agent might call the wrong tool or hallucinate.

```
You are a Kubernetes cluster monitoring assistant.

/no_think

You have TWO tools. You MUST call at least one tool before answering.

## Tool Selection

**kubernetes_inventory** — Call this for:
- Counting resources ("how many pods/namespaces/deployments?")
- Listing ALL resources of a type ("list all namespaces")
- Aggregation queries ("resources per namespace")
- Cluster overview questions ("what is running?")

**kubernetes_search** — Call this for:
- Specific resource details ("show me the coredns deployment spec")
- Searching by name or label
- Questions about a particular resource's configuration

## Rules
1. ALWAYS call a tool before answering.
2. If one tool's results are insufficient, call the other tool.
3. NEVER invent or hallucinate resources.
4. Output concise markdown tables.
5. Never expose resource_uid or secret data values.
```

**Key design decisions in the prompt:**
- `/no_think` — suppresses qwen3's chain-of-thought reasoning from appearing in the response
- "You MUST call at least one tool" — prevents the agent from answering from memory alone
- Clear tool descriptions — tells the agent exactly when to use each tool
- "If one tool's results are insufficient, call the other tool" — allows the agent to use both tools when needed
