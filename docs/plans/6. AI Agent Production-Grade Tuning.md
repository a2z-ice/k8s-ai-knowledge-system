# 6. AI Agent Production-Grade Tuning

## Date: 2026-03-07
## Branch: n8n-agent
## Status: Complete — All 12 E2E tests pass (1.2 min total)

---

## Overview

The AI chat endpoint (`/webhook/k8s-ai-chat/chat`) had three critical production issues:

1. **Thinking leak** — qwen3:8b dumped chain-of-thought reasoning ("Okay, let's see...") into every response
2. **Context overflow** — Qdrant payloads sent ~37K tokens to the LLM, exceeding its effective context window and causing Ollama to hang
3. **Incomplete results** — `topK: 10` returned too few vectors for aggregation queries like "pods in each namespace"

After tuning, response times dropped from **60s+ timeouts to 2-8 seconds**, responses are clean markdown tables, and all 12 E2E tests pass.

## Challenge 1: Thinking Leak (Chain-of-Thought in Response)

### Symptom

Every AI chat response started with internal reasoning text:

```
Okay, let's see... the user is asking about pods. Let me check the data.
From the tool response, there are entries for Pods in kube-system...
| Namespace | Pod Count |
|-----------|-----------|
| kube-system | 3 |
```

### Root Cause Analysis

**Layer 1 — Workflow JSON:** The workflow had `"think": false` in the Ollama Chat Model options:
```json
{
  "model": "qwen3:8b",
  "options": { "temperature": 0.1, "think": false }
}
```

**Layer 2 — n8n schema drops it:** The n8n Ollama Chat Model node v1 schema (`v1.schema.js`) defines only these options:
```
temperature, topK, topP, frequencyPenalty, keepAlive, lowVram, mainGpu,
numBatch, numCtx, numGpu, numPredict, numThread, penalizeNewline,
presencePenalty, repeatPenalty, useMLock, useMMap, vocabOnly, format
```
**`think` is NOT in this list.** n8n silently drops it during validation.

**Layer 3 — langchain/ollama supports it:** The `@langchain/ollama` library (v1.0.2) fully supports `think`:
```javascript
// @langchain/ollama/dist/chat_models.js line 446
invocationParams(options) {
  return {
    model: this.model,
    think: this.think,     // <-- supported here
    options: { ... }
  };
}
```

But n8n's `LmChatOllama.node.js` creates ChatOllama with `...options` spread, and since `think` was stripped by the schema, it's never passed:
```javascript
const model = new ollama_1.ChatOllama({
    ...options,   // <-- think is missing from options
    baseUrl: credentials.baseUrl,
    model: modelName,
});
```

**Layer 4 — qwen3 template behavior:** Without `think` set in the API call, the qwen3 chat template (`$.IsThinkSet` is false) does not append `/no_think` to user messages or prepend empty `<think></think>` blocks. The model defaults to thinking mode, putting chain-of-thought directly into the `content` field.

**Verification:** Direct Ollama API call confirms `think: false` works correctly:
```bash
curl http://192.168.1.154:11434/api/chat -d '{
  "model": "qwen3:8b",
  "messages": [{"role":"user","content":"Say hello"}],
  "think": false, "stream": false
}'
# Response: {"message":{"content":"Hello!"}}  <-- clean, no thinking
```

### Fix: Startup Patch in n8n Deployment

Since we cannot modify the n8n node schema (it's inside the container's read-only `node_modules`), the fix patches the file at container startup before n8n starts.

**File:** `infra/k8s/n8n/n8n-deployment.yaml`

```yaml
containers:
  - name: n8n
    image: n8nio/n8n:latest
    securityContext:
      runAsUser: 0          # Start as root to patch read-only node_modules
    command:
      - sh
      - -c
      - |
        # Patch n8n Ollama Chat Model node to pass think:false to ChatOllama
        OLLAMA_FILE=$(find /usr/local/lib/node_modules/n8n \
          -path "*/LMChatOllama/LmChatOllama.node.js" 2>/dev/null | head -1)
        if [ -n "$OLLAMA_FILE" ]; then
          sed -i "s/const model = new ollama_1.ChatOllama({/const model = new ollama_1.ChatOllama({ think: false,/" "$OLLAMA_FILE"
          echo "Patched think:false into $OLLAMA_FILE"
        fi
        # Drop to user node (1000) and start n8n via the standard entrypoint
        exec su -s /bin/sh node -c "exec /docker-entrypoint.sh"
```

**How it works:**
1. Container starts as `root` (runAsUser: 0)
2. `sed` patches `LmChatOllama.node.js` to inject `think: false` into the ChatOllama constructor
3. `su` drops privileges to user `node` (uid 1000)
4. n8n starts via the standard `/docker-entrypoint.sh`

**The patched line becomes:**
```javascript
const model = new ollama_1.ChatOllama({ think: false, ...options, baseUrl: ... });
```

**Verification log on pod startup:**
```
Patched think:false into /usr/local/lib/node_modules/n8n/node_modules/.pnpm/@n8n+n8n-nodes-langchain@.../LmChatOllama.node.js
Initializing n8n process
n8n ready on ::, port 5678
```

### Approaches Tried and Rejected

| Approach | Why it failed |
|----------|--------------|
| `"think": false` in workflow JSON options | n8n schema drops unknown options silently |
| `/no_think` tag in system prompt | qwen3 only respects this in the chat template via `$.IsThinkSet`, not in message content |
| Custom `qwen3:8b-nothink` Modelfile | Suppressed thinking but model ignored system prompt formatting rules (verbose output, missing keywords) |
| Init container to create patched file + subPath mount | Path contains version hash, too fragile across n8n upgrades |
| `kubectl exec sed` on running pod | File owned by root, container runs as uid 1000, permission denied |

## Challenge 2: Context Overflow (Ollama Hangs on Large Payloads)

### Symptom

Queries that triggered the Qdrant Vector Store tool caused Ollama to hang indefinitely. The n8n webhook never returned. Ollama became completely unresponsive (even `ollama ps` hung). Required `pkill ollama` to recover.

### Root Cause Analysis

**Measured payload sizes (BEFORE fix):**

| Metric | Value |
|--------|-------|
| 30 Qdrant results total size | **149,028 bytes** |
| Estimated token count | **~37,257 tokens** |
| qwen3:8b context window | 40,960 tokens |
| Per-result `raw_spec_json` size | **~4,443 chars** (~90% of each result) |

The `raw_spec_json` metadata field stored the **complete Kubernetes resource manifest** (containers, volumes, probes, env vars, etc.). For a Deployment, this is ~4.4KB of JSON per result. With 30 results + system prompt + tool-calling XML + chat history, the total exceeded qwen3:8b's context window.

**Example `raw_spec_json` for a Deployment (truncated):**
```json
{"replicas":1,"selector":{"matchLabels":{"app":"n8n"}},"template":{"spec":{
"containers":[{"name":"n8n","image":"n8nio/n8n:latest","ports":[{"containerPort":5678}],
"env":[{"name":"N8N_BASIC_AUTH_ACTIVE","value":"true"},{"name":"N8N_BASIC_AUTH_USER","value":"admin"},...],
"volumeMounts":[{"name":"n8n-data","mountPath":"/home/node/.n8n"}],
"readinessProbe":{"httpGet":{"path":"/healthz","port":5678},...},...}],
"initContainers":[{"name":"fix-permissions","image":"busybox","command":["sh","-c","chown -R 1000:1000 /home/node/.n8n"]}],
"hostAliases":[{"ip":"192.168.1.154","hostnames":["host.docker.internal"]}],...}}}
```

This full manifest is **useless for the LLM** — it needs to know "n8n deployment, 1 replica, image n8nio/n8n:latest" not the entire pod spec.

### Fix: Compact `spec_summary` in CDC Workflow

**File:** `workflows/n8n_cdc_k8s_flow.json` — `Format Document` node

Replaced `raw_spec_json` with a `buildSpecSummary()` function that extracts only the fields the LLM needs:

```javascript
function buildSpecSummary(kind, rawSpec) {
  const spec = JSON.parse(rawSpec);
  const parts = [];
  // Deployment/ReplicaSet/StatefulSet/DaemonSet
  if (spec.replicas !== undefined) parts.push(`replicas=${spec.replicas}`);
  // Pod containers
  const containers = spec.containers || spec.template?.spec?.containers || [];
  if (containers.length)
    parts.push(`containers=[${containers.map(c => `${c.name}:${c.image}`).join(', ')}]`);
  // Service
  if (kind === 'Service') { parts.push(`type=${spec.type}`); }
  if (spec.ports) parts.push(`ports=[${spec.ports.map(p => `${p.port}/${p.protocol}`).join(',')}]`);
  if (spec.clusterIP) parts.push(`clusterIP=${spec.clusterIP}`);
  // Secret
  if (kind === 'Secret') { parts.push(`type=${spec.type || 'Opaque'}`); }
  // PVC
  if (spec.resources?.requests?.storage) parts.push(`storage=${spec.resources.requests.storage}`);
  // Status
  if (spec.readyReplicas !== undefined) parts.push(`ready=${spec.readyReplicas}`);
  return parts.join('; ') || 'no-spec';
}
```

**Example output:** `replicas=1; containers=[n8n:n8nio/n8n:latest]` (~50 chars vs ~4,400 chars)

**Measured payload sizes (AFTER fix):**

| Metric | Before | After | Reduction |
|--------|--------|-------|-----------|
| 30 results total | 149,028 bytes | **39,108 bytes** | **74%** |
| Estimated tokens | ~37,257 | **~9,777** | **74%** |
| Per-result spec field | ~4,443 chars | **~50-200 chars** | **95-97%** |

### Additional Fix: Tuned Ollama Model

Created `qwen3:8b-k8s` with production parameters:

```bash
cat > /tmp/Modelfile.k8s <<'EOF'
FROM qwen3:8b
PARAMETER temperature 0
PARAMETER top_p 0.9
PARAMETER top_k 20
PARAMETER repeat_penalty 1.1
PARAMETER num_ctx 32768
PARAMETER num_predict 1024
EOF
ollama create qwen3:8b-k8s -f /tmp/Modelfile.k8s
```

| Parameter | Default (qwen3:8b) | Tuned (qwen3:8b-k8s) | Reason |
|-----------|--------------------|-----------------------|--------|
| temperature | 0.6 | **0** | Deterministic output; no creative variation needed for data extraction |
| num_ctx | 40960 | **32768** | Explicit context window; leaves headroom for KV cache |
| num_predict | unlimited | **1024** | Caps output length; prevents runaway verbose responses |
| repeat_penalty | 1.0 | **1.1** | Discourages repetitive phrasing in table output |
| top_p | 0.95 | **0.9** | Slightly narrower sampling for more focused output |

## Challenge 3: Incomplete Results (topK Too Low)

### Symptom

Query "pods in each namespace" returned only 3 pods from kube-system, missing k8s-ai (5+ pods), local-path-storage, and kube-node-lease namespaces.

### Root Cause

`topK: 10` in the Qdrant Vector Store node. With 80+ indexed resources across 10 resource types and 6 namespaces, 10 results returned a mix of types (Services, ConfigMaps, Pods) from only 1-2 namespaces.

### Fix

**File:** `workflows/n8n_ai_k8s_flow.json`

```
"topK": 10  -->  "topK": 30
```

This matches the E2E test 4 design which already uses `limit: 50` (documented in CLAUDE.md). With the compact `spec_summary` payloads, 30 results = ~10K tokens which fits comfortably in the 32K context.

## Challenge 4: System Prompt Optimization

### Symptom

Model produced inconsistent output: sometimes verbose multi-section markdown with headers, bullet points, and raw JSON dumps; sometimes concise tables.

### Fix

Replaced the verbose multi-paragraph prompt with 8 numbered rules:

```
You are a Kubernetes cluster monitoring assistant.
Every question is about the LIVE cluster. Always call the tool first.

Rules:
1. ALWAYS use kubernetes_knowledge_base tool before answering.
2. Answer ONLY from tool results. Never invent data.
3. Output a concise markdown table with columns: Kind | Name | Namespace | Details.
4. Include ALL matching resources from the tool results — do not omit any.
5. After the table, add at most one summary sentence.
6. Never output raw JSON, reasoning, or chain-of-thought.
7. Never expose resource_uid or secret data values.
8. If no results: | Status | No indexed Kubernetes resources found |
```

**Why numbered rules work better:** Small LLMs (8B) follow short, numbered instructions more reliably than prose paragraphs. Each rule is independently verifiable.
