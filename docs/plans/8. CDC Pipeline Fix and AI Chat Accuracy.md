# 8. CDC Pipeline Fix and Multi-Tool AI Agent

**Date:** 2026-03-07
**Status:** Complete

---

## Overview

The AI chat endpoint (`/webhook/k8s-ai-chat/chat`) returned incorrect resource counts and hallucinated resource names. Three root causes were identified and fixed:

1. **CDC pipeline race condition** — duplicate points in Qdrant from rapid k8s events
2. **AI Agent architecture flaw** — pre-loading all data bypassed tool-calling entirely
3. **Chat memory poisoning** — cached wrong answers replayed in subsequent queries

The solution is a **multi-tool AI Agent** that actively decides which tool to use for each query:
- `kubernetes_inventory` (Code Tool) — scrolls all Qdrant data for aggregation/counting
- `kubernetes_search` (Qdrant Vector Store) — semantic search for specific resource details

---

## Issue 1: CDC Pipeline Duplicate Points

### Symptoms

- Qdrant contained multiple points for the same k8s resource (same `resource_uid`)
- Pod counts were inflated (e.g., 8 pods shown as 12)

### Root Cause

The original CDC flow used n8n's native **Qdrant Vector Store insert** node, which auto-generates a random UUID as the point ID on every insertion. When a pod transitions through states (Pending → Running), multiple MODIFIED events create duplicate points.

### Fix

Replaced the n8n Qdrant Vector Store insert with HTTP Request nodes that use `resource_uid` as the Qdrant point ID:

```json
{
  "method": "PUT",
  "url": "http://qdrant:6333/collections/k8s/points",
  "jsonBody": "={{ JSON.stringify({ points: [{ id: resource_uid, vector: embedding, payload: {...} }] }) }}"
}
```

Using `resource_uid` as the point ID makes upserts **idempotent** — the same resource always overwrites the same point.

### Why HTTP Instead of Native Qdrant Node

The native n8n Qdrant Vector Store **insert** node (`mode: "insert"`) does NOT allow setting a custom point ID. It auto-generates a random UUID for every insertion. This is by design in n8n's LangChain integration — the native node is meant for document ingestion where deduplication isn't critical.

For CDC (Change Data Capture), **idempotent upserts are essential**. The only way to control the point ID is via Qdrant's REST API directly, hence the HTTP Request node.

The native Qdrant Vector Store node is still used in `retrieve-as-tool` mode for semantic search — that functionality works perfectly with native nodes.

### Verification

```bash
curl -s http://localhost:30001/collections/k8s/points/scroll \
  -X POST -H 'Content-Type: application/json' \
  -d '{"limit":500,"with_payload":{"include":["metadata"]}}' | \
  python3 -c "
import json, sys
points = json.load(sys.stdin)['result']['points']
uids = [p['payload']['metadata']['resource_uid'] for p in points]
print(f'Points: {len(points)}, Unique UIDs: {len(set(uids))}')
assert len(points) == len(set(uids)), 'FAIL: duplicates exist'
print('PASS: no duplicates')
"
```

## Issue 2: AI Agent Not Using Tools (Shortcut Architecture)

### Symptoms

- AI Agent never called the `kubernetes_search` Qdrant tool
- All answers came from pre-loaded context or chat memory
- The system didn't scale — every query loaded all 80+ resources into context

### Root Cause — The Shortcut

The previous "fix" pre-loaded ALL Qdrant data into the AI Agent's prompt:

```
Chat Trigger → Scroll All Resources → Build Inventory → AI Agent
                  (HTTP: scroll 200)    (Code: stuff into chatInput)
```

This had three fundamental problems:

1. **No tool-calling**: The AI Agent had all data in context, so it never called the Qdrant Vector Store tool. The tool was connected but useless.
2. **Doesn't scale**: Loading 200 resources into `chatInput` every query wastes context window and breaks with larger clusters.
3. **Not production-grade**: It's a brute-force shortcut — the agent acts as a text parser, not an intelligent tool-using system.

### The n8n chatTrigger Bypass

An additional complication: when using `promptType: "auto"` (default), the AI Agent node reads `chatInput` directly from the chatTrigger's session context, **not** from the incoming data item. The Build Inventory node's enriched `chatInput` was ignored. A `promptType: "define"` hack was needed to force the agent to read from the data pipeline — another indication this wasn't the right architecture.

### Fix — Multi-Tool AI Agent

Removed the shortcut entirely. The AI Agent now connects directly to the Chat Trigger with **two tools**:

```
Chat Trigger → AI Agent (toolsAgent, maxIterations=5)
                 ├── Tool 1: kubernetes_search  (Qdrant Vector Store, semantic, topK=20)
                 │     └── Embeddings Ollama (nomic-embed-text)
                 ├── Tool 2: kubernetes_inventory  (Code Tool, HTTP→Qdrant scroll)
                 ├── Ollama Chat Model (qwen3:14b-k8s)
                 └── Postgres Chat Memory
```

**Tool 1: `kubernetes_search`** — Native n8n Qdrant Vector Store in `retrieve-as-tool` mode. Uses semantic similarity to find relevant resources. Best for specific detail queries ("show me the coredns deployment spec").

**Tool 2: `kubernetes_inventory`** — n8n Code Tool (`toolCode`) that:
1. Calls Qdrant scroll API via `this.helpers.httpRequest()` to get ALL resources
2. Groups them by kind and namespace with counts
3. Returns a structured inventory text

The agent's system prompt instructs it when to use each tool:
- **Counting/listing/aggregation** → call `kubernetes_inventory`
- **Specific resource details** → call `kubernetes_search`

### Why Code Tool (not Tool Workflow)

The `toolWorkflow` node (typeVersion 2) requires a `workflowInputs` resourceMapper that's difficult to configure correctly via JSON import. In testing, it failed with "The workflow did not return a response" errors due to configuration mismatches.

The `toolCode` node is simpler and more reliable:
- No sub-workflow needed (no activation, no separate publish)
- Uses `this.helpers.httpRequest()` which is available in the n8n JavaScript sandbox
- The code runs inline within the AI Agent execution
- Returns a plain string directly

### Why Native Qdrant Node for Search

The Qdrant Vector Store node in `retrieve-as-tool` mode is used because:
- It handles embedding generation automatically (via Embeddings Ollama sub-node)
- It integrates natively with the AI Agent's tool-calling loop
- Semantic search is its designed use case
- No HTTP configuration needed — just set collection, topK, and credentials

### System Prompt

```
You are a Kubernetes cluster monitoring assistant for the kind cluster "k8s-ai".

/no_think

You have TWO tools. You MUST call at least one tool before answering.

## Tool Selection

**kubernetes_inventory** — Call this for:
- Counting resources ("how many pods/namespaces/deployments?")
- Listing ALL resources of a type
- Aggregation queries ("resources per namespace")
- Cluster overview questions

**kubernetes_search** — Call this for:
- Specific resource details ("show me the coredns deployment spec")
- Searching by name or label
- Questions about configuration, containers, or ports

## Rules
1. ALWAYS call a tool before answering. Never guess.
2. If one tool's results are insufficient, call the other tool.
3. NEVER invent or hallucinate resources, names, or counts.
4. Output concise markdown tables for structured data.
5. Never expose resource_uid or secret data values.
```

## Issue 3: Chat Memory Poisoning

### Root Cause

The Postgres Chat Memory (`contextWindowLength: 5`) cached wrong answers. The LLM saw its own previous (wrong) answers and repeated them — a self-reinforcing loop.

### Fix

Clear chat history after fixing data accuracy:

```bash
POSTGRES_POD=$(kubectl --context kind-k8s-ai -n k8s-ai get pod -l app=postgres -o jsonpath='{.items[0].metadata.name}')
kubectl --context kind-k8s-ai -n k8s-ai exec ${POSTGRES_POD} -- \
  psql -U n8n -d n8n_memory -c "TRUNCATE TABLE n8n_chat_histories;"
```

## Production-Grade Architecture

### Final AI K8s Flow

```
User query (Chat Trigger, public webhook)
  → AI Agent (toolsAgent, maxIterations=5)
      Decision: Which tool to call?
      ├── kubernetes_inventory (Code Tool)
      │     → HTTP POST qdrant:6333/collections/k8s/points/scroll (limit 200)
      │     → Group by kind/namespace → structured inventory text
      │     → Return to agent
      ├── kubernetes_search (Qdrant Vector Store, retrieve-as-tool)
      │     → Embed query (nomic-embed-text:latest, 768-dim)
      │     → Qdrant similarity search (topK=20)
      │     → Return matching resources with metadata
      ├── Ollama Chat Model (qwen3:14b-k8s, temp=0)
      └── Postgres Chat Memory (session: k8s-ai-global, contextWindow: 5)
  → Response (markdown table)
```

### Why This is Production-Grade

1. **Tool-using agent**: The LLM actively decides which tool to call based on the query. Not a brute-force pre-load.
2. **Separation of concerns**: Semantic search for detail queries, full scroll for aggregation — each tool does what it's best at.
3. **Scalable**: Only loads full inventory when the agent decides it's needed (aggregation queries). Detail queries use efficient vector search.
4. **Native n8n nodes**: Uses Qdrant Vector Store (retrieve-as-tool) natively. Only the inventory tool uses HTTP (because n8n doesn't have a "scroll all" node).
5. **Idempotent CDC**: `resource_uid` as point ID eliminates duplicates regardless of event ordering.
6. **Memory management**: PostgreSQL-backed chat memory with automatic context windowing.
7. **15 E2E tests**: Including 3 accuracy tests that compare AI output against real `kubectl` results.

### CDC K8s Flow (Final)

```
Kafka Trigger (k8s-resources)
  → Parse Message (embed_text construction)
  → Delete Existing Vector (HTTP → Qdrant filter by resource_uid)
  → Is Delete Event?
      → [TRUE]: stop (point already deleted)
      → [FALSE]: Format Document → Generate Embedding (HTTP → Ollama)
                  → Upsert to Qdrant (HTTP PUT with resource_uid as point ID)
```

## Lessons Learned

1. **Don't bypass the agent with pre-loaded context** — If you give the LLM all data in the prompt, it won't use tools. This defeats the purpose of having a tool-using agent and doesn't scale.

2. **n8n chatTrigger bypasses data pipeline for chatInput** — The AI Agent reads `chatInput` from the trigger's session, not from incoming data. Use `promptType: "define"` only if you must override this — but better to fix the architecture.

3. **Use `toolCode` over `toolWorkflow` for reliability** — The `toolCode` node with `this.helpers.httpRequest()` is simpler and avoids sub-workflow activation/configuration issues.

4. **Deterministic point IDs eliminate CDC races** — Using k8s resource UID as Qdrant point ID makes upserts idempotent.

5. **Chat memory creates feedback loops** — Always clear memory after fixing data accuracy.

6. **Test with exact names, not just counts** — Ask the model for exact resource names to immediately reveal hallucination.

7. **System prompt drives tool selection** — A clear system prompt with explicit tool routing rules is critical for correct tool-calling behavior.
