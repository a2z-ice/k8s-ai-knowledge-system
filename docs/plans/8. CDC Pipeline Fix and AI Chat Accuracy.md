# 8. CDC Pipeline Fix and AI Chat Accuracy

**Date:** 2026-03-07
**Status:** Complete

---

## Overview

The AI chat endpoint (`/webhook/k8s-ai-chat/chat`) returned incorrect resource counts. Two root causes were identified and fixed:

1. **CDC pipeline race condition** — duplicate points in Qdrant from rapid k8s events
2. **AI Agent ignoring enriched context** — n8n chatTrigger bypassed the Build Inventory node

A third issue — **chat memory poisoning** — amplified incorrect answers by caching and replaying them.

---

## Issue 1: CDC Pipeline Duplicate Points

### Symptoms

- Qdrant contained multiple points for the same k8s resource (same `resource_uid`)
- Pod counts were inflated (e.g., 8 pods shown as 12)
- After a reset+resync, counts were correct temporarily but drifted as pods restarted

### How to Identify

```bash
# Check for duplicate resource_uids in Qdrant
curl -s http://localhost:30001/collections/k8s/points/scroll \
  -X POST -H 'Content-Type: application/json' \
  -d '{"limit":500,"with_payload":{"include":["metadata"]}}' | \
  python3 -c "
import json, sys
d = json.load(sys.stdin)
points = d['result']['points']
uids = {}
for p in points:
    uid = p.get('payload',{}).get('metadata',{}).get('resource_uid','?')
    uids.setdefault(uid, []).append(p['id'])
dupes = {u: ids for u, ids in uids.items() if len(ids) > 1}
if dupes:
    print(f'DUPLICATES FOUND: {len(dupes)} resources have multiple points')
    for uid, ids in list(dupes.items())[:5]:
        print(f'  {uid}: {len(ids)} points (IDs: {ids})')
else:
    print(f'No duplicates found across {len(points)} points')
"
```

### Root Cause

The original CDC flow used n8n's native **Qdrant Vector Store insert** node, which auto-generates a random UUID as the point ID on every insertion. When a pod transitions through states (Pending → Running), multiple MODIFIED events arrive rapidly:

```
Event 1 (MODIFIED): resource_uid=abc → Qdrant point ID = random-uuid-1
Event 2 (MODIFIED): resource_uid=abc → Qdrant point ID = random-uuid-2  ← DUPLICATE
```

The delete-before-insert used `metadata.resource_uid` filter, but a race condition between concurrent n8n executions meant event 2's delete could run before event 1's insert completed.

### Fix

Replaced the n8n Qdrant Vector Store insert node with two HTTP Request nodes that use `resource_uid` as the Qdrant point ID directly:

**Generate Embedding** (HTTP Request → Ollama):
```json
{
  "method": "POST",
  "url": "http://host.docker.internal:11434/api/embeddings",
  "jsonBody": "={{ JSON.stringify({ model: 'nomic-embed-text:latest', prompt: $json.pageContent }) }}"
}
```

**Upsert to Qdrant** (HTTP Request → Qdrant PUT):
```json
{
  "method": "PUT",
  "url": "http://qdrant:6333/collections/k8s/points",
  "jsonBody": "={{ JSON.stringify({ points: [{ id: $('Format Document').first().json.metadata.resource_uid, vector: $json.embedding, payload: { pageContent: ..., metadata: ... } }] }) }}"
}
```

Using `resource_uid` as the point ID makes upserts **idempotent** — the same resource always overwrites the same point, regardless of how many events arrive.

### Verification

```bash
# After fix, every resource_uid should map to exactly one point
curl -s http://localhost:30001/collections/k8s/points/scroll \
  -X POST -H 'Content-Type: application/json' \
  -d '{"limit":500,"with_payload":{"include":["metadata"]}}' | \
  python3 -c "
import json, sys
points = json.load(sys.stdin)['result']['points']
uids = [p['payload']['metadata']['resource_uid'] for p in points]
print(f'Points: {len(points)}, Unique UIDs: {len(set(uids))}')
assert len(points) == len(set(uids)), 'FAIL: duplicates exist'
print('PASS: no duplicates')
"
```

## Issue 2: AI Agent Ignoring Enriched Context

### Symptoms

- AI chat returned completely fabricated pod names (e.g., `coredns-5644d7b6d9-2jqhk` instead of `coredns-7d764666f9-4cwvr`)
- Pod counts were consistently wrong in the same way (e.g., "default: 5, kube-system: 20, ai-namespace: 10")
- The model worked perfectly when called directly via Ollama API with the same inventory data

### How to Identify

```bash
# Ask the model to list exact pod names — if they don't match kubectl, the context isn't reaching the model
curl -s -X POST http://localhost:30000/webhook/k8s-ai-chat/chat \
  -H 'Content-Type: application/json' \
  -d '{"chatInput": "List the exact names of all pods in the kube-system namespace"}' | \
  python3 -c "import json,sys; print(json.load(sys.stdin).get('output',''))"

# Compare with actual kubectl output
kubectl --context kind-k8s-ai get pods -n kube-system --no-headers -o custom-columns=NAME:.metadata.name
```

If the pod names from the chat don't match kubectl output, the enriched context isn't reaching the LLM.

### Root Cause

The n8n AI Agent node (`typeVersion: 1.7`) with a **chatTrigger** reads `chatInput` directly from the trigger's session context, **not** from the incoming data item. The architecture was:

```
Chat Trigger → Scroll All Resources → Build Inventory → AI Agent
                                          ↓
                               output: { chatInput: enrichedInput }
```

Build Inventory correctly enriched `chatInput` with `[CLUSTER INVENTORY]` data, but the AI Agent ignored it and used the original `chatInput` from the Chat Trigger instead. The model had no inventory data and hallucinated all resource names and counts.

### Fix

Added `promptType: "define"` and `text: "={{ $json.chatInput }}"` to the AI Agent node parameters:

```json
{
  "parameters": {
    "agent": "toolsAgent",
    "promptType": "define",
    "text": "={{ $json.chatInput }}",
    "options": {
      "systemMessage": "..."
    }
  }
}
```

With `promptType: "define"`, the AI Agent evaluates the `text` expression against the incoming data item (from Build Inventory), which contains the enriched chatInput with the full cluster inventory.

### Verification

```bash
# Clear memory first
POSTGRES_POD=$(kubectl --context kind-k8s-ai -n k8s-ai get pod -l app=postgres -o jsonpath='{.items[0].metadata.name}')
kubectl --context kind-k8s-ai -n k8s-ai exec ${POSTGRES_POD} -- \
  psql -U n8n -d n8n_memory -c "TRUNCATE TABLE n8n_chat_histories;"

# Test — counts should match kubectl
curl -s -X POST http://localhost:30000/webhook/k8s-ai-chat/chat \
  -H 'Content-Type: application/json' \
  -d '{"chatInput": "How many pods in each namespace?"}' | \
  python3 -c "import json,sys; print(json.load(sys.stdin).get('output',''))"

# Verify against reality
kubectl --context kind-k8s-ai get pods --all-namespaces --no-headers | awk '{print $1}' | sort | uniq -c
```

## Issue 3: Chat Memory Poisoning

### Symptoms

- Even after fixing the underlying data issues, the AI chat continued returning the same wrong answers
- Clearing the Qdrant data and re-querying still produced stale results
- The wrong answers were suspiciously consistent (exact same numbers every time)

### How to Identify

```bash
# Check how many messages are cached
POSTGRES_POD=$(kubectl --context kind-k8s-ai -n k8s-ai get pod -l app=postgres -o jsonpath='{.items[0].metadata.name}')
kubectl --context kind-k8s-ai -n k8s-ai exec ${POSTGRES_POD} -- \
  psql -U n8n -d n8n_memory -c "SELECT COUNT(*) FROM n8n_chat_histories WHERE session_id = 'k8s-ai-global';"

# View recent messages
kubectl --context kind-k8s-ai -n k8s-ai exec ${POSTGRES_POD} -- \
  psql -U n8n -d n8n_memory -c "SELECT session_id, LEFT(message::text, 100) FROM n8n_chat_histories ORDER BY id DESC LIMIT 10;"
```

### Root Cause

The Postgres Chat Memory node (`contextWindowLength: 5`) caches the last 5 message pairs. When the AI returned wrong answers, those wrong answers were stored in memory. On subsequent queries, the LLM saw its own previous (wrong) answers and repeated them — a self-reinforcing loop.

### Fix

Clear the chat history table after fixing the underlying data accuracy issues:

```bash
POSTGRES_POD=$(kubectl --context kind-k8s-ai -n k8s-ai get pod -l app=postgres -o jsonpath='{.items[0].metadata.name}')
kubectl --context kind-k8s-ai -n k8s-ai exec ${POSTGRES_POD} -- \
  psql -U n8n -d n8n_memory -c "TRUNCATE TABLE n8n_chat_histories;"
```

## Architecture After Fix

### AI K8s Flow (updated)

```
User query (Chat Trigger)
  → Scroll All Resources (HTTP → Qdrant scroll API, limit 200)
  → Build Inventory (Code: group by kind/namespace, enrich chatInput)
  → AI Agent (promptType: "define", text: "={{ $json.chatInput }}")
      → Ollama Chat Model (qwen3:14b-k8s, temp=0)
      → Qdrant Vector Store (semantic search tool, topK=20)
          → Embeddings Ollama (nomic-embed-text)
      → Postgres Chat Memory (session: k8s-ai-global)
  → Response
```

The key difference: the AI Agent now receives the **enriched chatInput** containing the full cluster inventory, enabling accurate aggregation queries (counts, listings) without relying on semantic search.

### CDC K8s Flow (updated)

```
Kafka Trigger (k8s-resources)
  → Parse Message (embed_text construction)
  → Delete Existing Vector (HTTP → Qdrant filter by resource_uid)
  → Is Delete Event? (If node)
      → [TRUE]: stop (point already deleted)
      → [FALSE]: Format Document → Generate Embedding (HTTP → Ollama)
                  → Upsert to Qdrant (HTTP PUT with resource_uid as point ID)
```

The key difference: uses `resource_uid` as the Qdrant point ID for **deterministic upserts**, eliminating race condition duplicates.

## Lessons Learned

1. **n8n chatTrigger bypasses data pipeline for chatInput** — The AI Agent node with a chatTrigger reads `chatInput` from the trigger's session, not from the incoming data item. Use `promptType: "define"` with an explicit expression to override this behavior.

2. **n8n expressions don't work in `options.systemMessage`** — The AI Agent's `systemMessage` field does not evaluate `={{ }}` expressions. Injecting dynamic data must happen via the `text` parameter or by enriching `chatInput` upstream.

3. **Deterministic point IDs eliminate CDC races** — Using the k8s resource UID as the Qdrant point ID makes upserts idempotent, regardless of event ordering or concurrency.

4. **Chat memory creates feedback loops** — When the LLM produces wrong answers, cached memory causes it to repeat them. Always clear memory after fixing data accuracy issues.

5. **Test with exact names, not just counts** — Asking the model for exact resource names (which are unique hash-based strings) immediately reveals hallucination. Aggregate counts can coincidentally match even when the model is fabricating data.
