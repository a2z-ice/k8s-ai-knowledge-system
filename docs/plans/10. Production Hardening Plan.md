# 10. Production Hardening Plan

**Date:** 2026-03-08
**Status:** Planned

---

## Overview

Comprehensive audit of the entire system (CDC pipeline, AI Agent, k8s-watcher, infrastructure, tests) identified improvements across data consistency, error handling, security, observability, and scalability. This plan prioritizes fixes by impact and implements the critical ones.

---

## P0 — Critical (Data Correctness & Security)

### 1. CDC Delete-Before-Upsert Race Condition

**Problem:** The CDC flow deletes the existing Qdrant point BEFORE generating the embedding. If Ollama is slow/down or Qdrant upsert fails, the resource disappears from the index with no recovery.

**Current flow:**
```
Parse Message → Delete Existing Vector → Is Delete? → Format → Embed → Upsert
```

**Fix:** Move the delete into a "delete-only" branch. For ADDED/MODIFIED events, use PUT (upsert) directly — Qdrant overwrites the point with the same `resource_uid` ID automatically.

**Fixed flow:**
```
Parse Message → Is Delete Event?
  → [TRUE]:  Delete Vector → Stop
  → [FALSE]: Format Document → Generate Embedding → Upsert to Qdrant
```

**Why this works:** The Qdrant PUT `/collections/k8s/points` with a known point ID is idempotent — it overwrites the existing point. No need to delete before upserting. The delete is only needed for actual DELETE events from Kubernetes.

**Impact:** Eliminates data loss when Ollama or Qdrant is temporarily unavailable during MODIFIED events.

---

### 2. CDC No Error Handling / Retry

**Problem:** HTTP nodes for Ollama embedding and Qdrant upsert have no retry logic. A single timeout or 5xx response silently drops the resource from the pipeline.

**Fix:** Add `retry` options to the HTTP Request nodes:

```json
{
  "options": {
    "retry": {
      "enabled": true,
      "maxRetries": 3,
      "retryInterval": 2000
    }
  }
}
```

n8n HTTP Request node v4 supports built-in retry with configurable count and interval.

**Impact:** Transient Ollama/Qdrant failures no longer cause permanent data loss.

---

### 3. Hardcoded Credentials in Manifests

**Problem:** Postgres password (`n8n`), n8n encryption key, and other secrets are plaintext in YAML manifests (deployment env vars).

**Current state (example from n8n deployment):**
```yaml
- name: DB_POSTGRESDB_PASSWORD
  value: "n8n"
```

**Fix:** Create Kubernetes Secret resources and reference them:
```yaml
- name: DB_POSTGRESDB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: n8n-secrets
      key: db-password
```

**Scope:** This is a kind cluster for development/demo. The fix improves hygiene but is not as urgent as it would be in a real production cluster.

---

### 4. n8n Runs as Root

**Problem:** n8n deployment has `securityContext.runAsUser: 0`. This is required because the n8n data directory (`/home/node/.n8n`) is mounted from a hostPath PV owned by root.

**Fix:** Use an initContainer to `chown` the data directory (already done for Kafka), then set `runAsUser: 1000` (node user):

```yaml
initContainers:
  - name: fix-permissions
    image: busybox
    command: ["sh", "-c", "chown -R 1000:1000 /home/node/.n8n"]
    volumeMounts:
      - name: n8n-data
        mountPath: /home/node/.n8n
securityContext:
  runAsUser: 1000
  runAsGroup: 1000
```

## P1 — High (Reliability)

### 5. k8s-watcher ClusterRole Too Broad

**Problem:** ClusterRole grants `get/list/watch` on Secrets across ALL namespaces. The watcher only needs access within the cluster it monitors.

**Assessment:** In a kind cluster where the watcher monitors the entire cluster, ClusterRole is actually correct — it needs to watch resources in all namespaces. However, the Secret access should ideally be read-only metadata (which it already is in code — `obj_to_payload` strips data values). The RBAC is appropriate for the use case.

**Recommendation:** Document this as an intentional design decision. In a multi-tenant production cluster, scope to specific namespaces using Role + RoleBinding per namespace.

---

### 6. Add Liveness and Readiness Probes

**Problem:** No pods have health checks. Kubernetes can't detect hung processes or failed startups.

**Fix:** Add probes to all deployments:

| Deployment | Liveness | Readiness |
|-----------|----------|-----------|
| n8n | `httpGet /healthz :5678` | `httpGet /healthz :5678` |
| Qdrant | `httpGet / :6333` | `httpGet /collections :6333` |
| Postgres | `exec pg_isready -U n8n` | `exec pg_isready -U n8n` |
| k8s-watcher | `httpGet /healthz :8080` | `httpGet /healthz :8080` |

---

### 7. Add Resource Limits

**Problem:** No CPU/memory requests or limits. A single pod can starve others on the kind node.

**Fix:** Add resource specifications:

| Pod | Request (CPU/Mem) | Limit (CPU/Mem) |
|-----|-------------------|-----------------|
| n8n | 200m / 512Mi | 1000m / 2Gi |
| Qdrant | 100m / 256Mi | 500m / 1Gi |
| Postgres | 100m / 128Mi | 500m / 512Mi |
| Kafka | 200m / 512Mi | 1000m / 2Gi |
| k8s-watcher | 50m / 64Mi | 200m / 256Mi |

## P2 — Medium (Observability & Scalability)

### 8. kubernetes_inventory Scroll Pagination

**Problem:** `limit: 200` is hardcoded. Clusters with >200 resources silently truncate results.

**Fix:** Implement pagination using Qdrant's `offset` field:

```javascript
let allPoints = [];
let offset = null;
do {
  const body = { limit: 100, with_payload: { include: ['metadata'] } };
  if (offset) body.offset = offset;
  const response = await this.helpers.httpRequest({
    method: 'POST',
    url: 'http://qdrant:6333/collections/k8s/points/scroll',
    body, json: true
  });
  allPoints.push(...(response.result?.points || []));
  offset = response.result?.next_page_offset || null;
} while (offset);
```

**Impact:** Supports clusters of any size without silent data loss.

---

### 9. CDC Pipeline Monitoring

**Problem:** No visibility into CDC health — no way to know if events are being processed, failing, or backed up.

**Fix options (lightweight, no external tools):**
- Add a Code node after Upsert that logs to stdout: `console.log(`CDC: ${kind}/${name} upserted`)`
- k8s-watcher already logs to stdout; ensure these are visible via `kubectl logs`
- Add a Qdrant points_count check to the `/healthz` endpoint of k8s-watcher

---

### 10. Negative E2E Tests

**Problem:** All tests assume happy path. No tests verify behavior when Ollama is slow, Qdrant is down, or Kafka is unavailable.

**Recommendation:** Add tests for:
- AI chat response when Qdrant is empty (should return "no resources found")
- AI chat timeout handling (Ollama slow response)
- CDC recovery after temporary Qdrant outage
