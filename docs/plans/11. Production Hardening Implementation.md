## Issue 1: CDC Delete-Before-Upsert Race Condition

### Problem

The CDC pipeline deleted the existing Qdrant point BEFORE generating a new embedding. If Ollama was slow, timed out, or returned an error, the resource disappeared from Qdrant permanently — with no recovery.

**Before (broken flow):**
```
Kafka Trigger → Parse Message → Delete Existing Vector → Is Delete Event?
                                                           → [TRUE]: stop
                                                           → [FALSE]: Format → Embed → Upsert
```

Every event (ADDED, MODIFIED, DELETED) first deleted the old vector. For MODIFIED events, if the embed/upsert failed after the delete, the resource was gone.

### Fix

Moved the `Is Delete Event?` check BEFORE the delete. Now only actual DELETE events trigger a vector deletion. For ADDED/MODIFIED events, the upsert overwrites the existing point via the same `resource_uid` point ID — Qdrant's PUT is idempotent.

**After (fixed flow):**
```
Kafka Trigger → Parse Message → Is Delete Event?
                                  → [TRUE]:  Delete Existing Vector (stop)
                                  → [FALSE]: Format Document → Generate Embedding → Upsert to Qdrant
```

### File Changed

**`workflows/n8n_cdc_k8s_flow.json`**

Connections reordered:
- `Parse Message` → now connects to `Is Delete Event?` (was: `Delete Existing Vector`)
- `Is Delete Event?` TRUE branch → `Delete Existing Vector` (was: empty/stop)
- `Is Delete Event?` FALSE branch → `Format Document` (unchanged)

Node positions swapped:
- `Is Delete Event?` moved to position [640, 300] (was [860, 300])
- `Delete Existing Vector` moved to position [860, 180] (was [640, 300])

## Issue 2: CDC No Error Handling / Retry

### Problem

The HTTP Request nodes for Ollama embedding and Qdrant upsert had no retry logic. A single transient error (Ollama timeout, Qdrant 503) silently dropped the resource from the pipeline. The CDC flow processed the Kafka message as "done" even though the vector was never stored.

### Fix

Added n8n node-level retry configuration to both HTTP nodes.

### File Changed

**`workflows/n8n_cdc_k8s_flow.json`**

**Generate Embedding node** — added:
```json
{
  "retryOnFail": true,
  "maxTries": 3,
  "waitBetweenTries": 2000,
  "options": {
    "timeout": 30000
  }
}
```

**Upsert to Qdrant node** — added:
```json
{
  "retryOnFail": true,
  "maxTries": 3,
  "waitBetweenTries": 2000,
  "options": {
    "timeout": 10000
  }
}
```

Each node now retries up to 3 times with 2-second intervals before failing. The embedding node has a 30-second timeout (Ollama can be slow under load); the upsert node has a 10-second timeout.

## Issue 3: kubernetes_inventory Hardcoded Scroll Limit

### Problem

The `kubernetes_inventory` Code Tool used a single Qdrant scroll call with `limit: 200`. Clusters with more than 200 resources would silently truncate results — the agent would report incomplete counts without any indication of missing data.

### Fix

Replaced the single scroll call with a pagination loop using Qdrant's `next_page_offset` field. The tool now fetches pages of 100 until all points are retrieved.

### File Changed

**`workflows/n8n_ai_k8s_flow.json`** — `kubernetes_inventory` Code Tool (`jsCode`)

**Before:**
```javascript
const response = await this.helpers.httpRequest({
  method: 'POST',
  url: 'http://qdrant:6333/collections/k8s/points/scroll',
  body: { limit: 200, with_payload: { include: ['metadata'] } },
  json: true
});
const points = response.result?.points || [];
```

**After:**
```javascript
let allPoints = [];
let offset = null;
do {
  const body = { limit: 100, with_payload: { include: ['metadata'] } };
  if (offset) body.offset = offset;
  const response = await this.helpers.httpRequest({
    method: 'POST',
    url: 'http://qdrant:6333/collections/k8s/points/scroll',
    body, json: true
  });
  const page = response.result?.points || [];
  allPoints.push(...page);
  offset = response.result?.next_page_offset || null;
} while (offset);
```

## Issue 4: No Liveness Probes on Deployments

### Problem

All deployments had readiness probes but no liveness probes. If a pod's process hung (accepting no new requests but not crashing), Kubernetes could not detect or recover from it. The pod would stay in `Running` state indefinitely with no traffic being served.

### Fix

Added `livenessProbe` alongside the existing `readinessProbe` on all 4 application deployments.

### Files Changed

**`infra/k8s/n8n/n8n-deployment.yaml`** — added:
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 5678
  initialDelaySeconds: 30
  periodSeconds: 15
  failureThreshold: 3
```

**`infra/k8s/qdrant/qdrant-deployment.yaml`** — added:
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 6333
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3
```

**`infra/k8s/postgres/postgres-deployment.yaml`** — added:
```yaml
livenessProbe:
  exec:
    command: ["pg_isready", "-U", "n8n", "-d", "n8n_memory"]
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3
```

**`infra/k8s/k8s-watcher/k8s-watcher-deployment.yaml`** — added:
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3
```

### Design Decisions

- **Liveness `initialDelaySeconds` is higher than readiness** — gives the process time to fully start before Kubernetes begins checking for hangs.
- **Liveness `failureThreshold: 3`** — tolerates brief unresponsiveness (e.g., during garbage collection) before restarting the pod.
- **Readiness probes unchanged** — they control traffic routing, liveness probes control restart behavior. Both are needed.

## Issue 5: Qdrant Deployment Strategy

### Problem (discovered during rollout)

Applying the liveness probe change triggered a rolling update of the Qdrant deployment. The default `RollingUpdate` strategy creates the new pod before terminating the old one. Qdrant uses WAL (Write-Ahead Log) file locks — two Qdrant instances cannot run on the same PersistentVolume simultaneously. The new pod crashed in a `CrashLoopBackOff` with:

```
Failed to load local shard: Wal error: Can't init WAL: Os { code: 11, kind: WouldBlock }
```

### Fix

Changed Qdrant's deployment strategy from `RollingUpdate` to `Recreate`. This ensures the old pod is fully terminated before the new one starts.

### File Changed

**`infra/k8s/qdrant/qdrant-deployment.yaml`** — added:
```yaml
spec:
  strategy:
    type: Recreate
```

## Issue 6: AI Agent Returning Incomplete Results for Follow-Up Queries

### Problem

When the user asked a sequence of questions:
1. "How many configmaps in k8s-ai?" → **Correct: 2**
2. "Show me the configmap data of this namespace" → **Wrong: only 1 shown (kube-root-ca.crt)**
3. "Give me those 2 configmap names" → **Wrong: only 1 shown**

**Root cause:** The agent used `kubernetes_search` (semantic search) for follow-up listing queries. Semantic search embeds the query and returns top-K by cosine similarity — it doesn't guarantee returning ALL matching resources. The query "show me configmaps" might rank one ConfigMap higher than another based on embedding similarity, missing the second one entirely.

Additionally, chat memory cached the incomplete answer from query 2, and the agent replayed it for query 3 without calling a tool.

### Fix — System Prompt Strengthened

The system prompt was rewritten to explicitly route ALL listing/showing/naming queries to `kubernetes_inventory` and to require a fresh tool call for every answer.

### File Changed

**`workflows/n8n_ai_k8s_flow.json`** — AI Agent `systemMessage`

Key changes:
1. `kubernetes_inventory` description expanded: added "show me the configmaps", "what configmaps exist", "give me the names", and "follow-up questions about resources previously counted or listed"
2. `kubernetes_search` scoped down: "Use this ONLY when the question is about ONE specific named resource"
3. Added rule: "When in doubt between tools, use kubernetes_inventory — it returns ALL resources and is always complete"
4. Added rule: "If the user previously counted N resources and now asks to list them, call kubernetes_inventory again — do NOT answer from memory"
5. Changed header from "never guess or use memory alone" to "never rely on chat history or memory alone"

## Issue 7: kubernetes_inventory Output Missing Per-Resource Details

### Problem

The inventory tool listed resources as comma-separated names:
```
ConfigMap (2 total):
  k8s-ai (2): pgadmin-servers, kube-root-ca.crt
```

This gave the agent no details to answer "show me the data" questions — it had to fall back to `kubernetes_search`, which returned incomplete results (Issue 6).

### Fix

Changed the inventory output to show each resource on its own line with spec details:
```
ConfigMap (2 total):
  k8s-ai (2):
    - pgadmin-servers [keys=[servers.json]; servers.json={...}]
    - kube-root-ca.crt [keys=[ca.crt]; ca.crt=-----BEGIN CERTIFICATE-----...]
```

### File Changed

**`workflows/n8n_ai_k8s_flow.json`** — `kubernetes_inventory` Code Tool (`jsCode`)

Changed the grouping to include labels and spec per resource:
```javascript
// Before
byKind[kind].push({ name, namespace, spec });
// ...
lines.push(`  ${ns} (${count}): ${names.join(', ')}`);

// After
byKind[kind].push({ name, namespace, labels, spec });
// ...
lines.push(`  ${ns} (${count}):`);
for (const item of nsItems) {
  lines.push(`    - ${item.name} [${item.spec}]`);
}
```

## Issue 8: ConfigMap Data Not Captured by k8s-watcher

### Problem

ConfigMaps have no `spec` field — their content is in the `data` field. The k8s-watcher's `obj_to_payload` function did `raw.get("spec") or {}`, which always returned `{}` for ConfigMaps. This meant:
- `embed_text` contained `Spec: {}` — no useful content for semantic search
- `raw_spec_json` was `{}` — no data for the agent to read
- `spec_summary` was `no-spec` — no summary in the inventory view

The AI agent literally had no ConfigMap data to show the user.

### Fix

Added ConfigMap-specific handling in `obj_to_payload`, similar to the existing Secret handling. For ConfigMaps, the `data` field is captured with values truncated to 200 characters to keep Kafka messages manageable.

### File Changed

**`k8s-watcher/watcher.py`** — `obj_to_payload` function

Added before the Secret handling block:
```python
# ConfigMap: capture data field (ConfigMaps have no spec)
if kind == "ConfigMap":
    cm_data = raw.get("data") or {}
    # Truncate large values to keep payload manageable
    spec = {
        k: (v[:200] + "..." if len(v) > 200 else v)
        for k, v in cm_data.items()
    }
```

This produces:
- `raw_spec_json`: `{"servers.json": "{\"Servers\": {\"1\": {\"Name\": \"k8s-ai Postgres\"...}}"}`
- `embed_text`: `Kubernetes ConfigMap named pgadmin-servers in namespace k8s-ai. Labels: none. Spec: {"servers.json": "..."}`

**`workflows/n8n_cdc_k8s_flow.json`** — `buildSpecSummary` function in Format Document node

Added ConfigMap handling:
```javascript
// ConfigMap
if (kind === 'ConfigMap') {
  const keys = Object.keys(spec);
  if (keys.length) parts.push(`keys=[${keys.join(',')}]`);
  for (const k of keys.slice(0, 3)) {
    const v = String(spec[k] || '').substring(0, 100);
    parts.push(`${k}=${v}${spec[k]?.length > 100 ? '...' : ''}`);
  }
}
```

This produces `spec_summary`: `keys=[servers.json]; servers.json={"Servers": {"1": {"Name": "k8s-ai Postgres"...`

### Deployment Steps

Since k8s-watcher is a Python service running as a Docker image inside kind:
1. `docker build -t k8s-watcher:latest ./k8s-watcher/`
2. `kind load docker-image k8s-watcher:latest --name k8s-ai`
3. `kubectl rollout restart deployment/k8s-watcher`
4. `POST /webhook/k8s-reset` to re-index all resources with the new data format

## Issue 10: Inventory Namespace Summary Showed Only Totals — LLM Confused Total Resources with Specific Kind Counts

### Problem

When the user asked "show me all namespaces", the agent reported `k8s-ai: 6` instead of `k8s-ai: 48`. Then when asked "give me detail about k8s-ai", the breakdown showed 47 resources across 7 kinds. The numbers were inconsistent.

**Root cause:** The original inventory output grouped by **kind first**, then namespace:
```
Pod (6 total):
  k8s-ai (6): ...
```

The agent saw `Pod (6 total)` under `k8s-ai` and reported 6 as the namespace total, ignoring all other kinds (Deployment, Service, ConfigMap, etc.).

After the namespace summary was added (Issue 7), it showed:
```
k8s-ai: 48 resources (ConfigMap:2, Deployment:5, Pod:7, ...)
```

But the LLM still read the total (48) for specific-kind queries like "how many pods" — a different failure mode.

### Fix

Restructured the namespace summary as a **markdown table with per-kind columns**:

```
NAMESPACE SUMMARY:
| Namespace | Total | Pod | Deployment | Service | ConfigMap | Secret | ReplicaSet | StatefulSet | DaemonSet | PVC | Other |
| k8s-ai | 48 | 7 | 5 | 10 | 2 | 0 | 19 | 1 | 0 | 4 | 0 |
| kube-system | 21 | 8 | 1 | 2 | 5 | 3 | 1 | 0 | 0 | 0 | 1 |
```

This gives the LLM a clear, unambiguous structure:
- "How many pods in each namespace?" → read the **Pod** column
- "Show me all namespaces" → read all rows with the **Total** column
- "How many resources in k8s-ai?" → read the **Total** column for `k8s-ai`

Also added system prompt rule: "Read the tool output carefully — the NAMESPACE SUMMARY shows totals AND per-kind breakdowns. When asked about a specific kind (e.g. pods), report the count for THAT kind, not the namespace total."

### File Changed

**`workflows/n8n_ai_k8s_flow.json`** — `kubernetes_inventory` Code Tool (`jsCode`) and system prompt

**Namespace summary output format changed:**
```javascript
// Before: single line per namespace with inline breakdown
lines.push(`  ${ns}: ${total} resources (${breakdown})`);

// After: markdown table with explicit per-kind columns
lines.push('| Namespace | Total | Pod | Deployment | Service | ConfigMap | ... |');
lines.push(`| ${ns} | ${total} | ${pod} | ${deployment} | ... |`);
```
